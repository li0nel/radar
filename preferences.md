# Radar Preferences

## Editorial Voice
- Dense and technical. Write like a Principal Engineer's internal notes.
- No preamble, no "in recent years..." — lead with the insight.
- Assume the reader has deep engineering context. Don't explain basics.
- Every insight ends with a "so what for me" line tied to my current projects and goals.
- Terse > verbose. If it can be said in one sentence, don't use three.

## User Context (for relevance scoring)
- Principal/Staff Software Engineer, targeting CTO/VP Engineering track
- Active interview processes (system design, Go coding)
- Building Spark (autonomous research/ideation agents)
- Interested in construction/property SaaS in ANZ market
- Heavy Claude Code user with sophisticated setup (hooks, skills, subagents)
- Solo developer / indie hacker
- Based in Brisbane, Australia

## Scoring Criteria
Weight these when deciding what to include:

1. **Novelty of insight** (highest weight): Does this say something I haven't seen before? A new framing, a counterintuitive finding, a production lesson? If it's a rehash of known patterns, skip it.
2. **Actionability**: Can I use this in the next 2 weeks? In my interview prep, in Spark, in my Claude Code setup, in a side project? If it's purely theoretical, skip it.
3. **Source authority**: Who is saying this and why should I trust them? Production experience > credentials > follower count.
4. **Specificity**: Concrete examples, code, numbers, architectures > vague claims and hand-waving.

### Untrusted
- Generic "top 10 AI tools" listicles

- Content that's primarily promotional for a specific product

## Anti-patterns (always filter these out)
- Hype without substance ("X will change everything")
- Just-in-case information (interesting but no clear application to my work)
- Rehashed basics (explanations of well-known patterns I already understand)
- Framework announcements without adoption evidence
- Thought leadership that's actually content marketing

## Format Preferences
- Code examples: yes, when they illustrate a pattern concisely
- Diagrams: yes, for architecture and system design topics
- Link to primary source: always
- Length per insight in living document: 1-3 dense paragraphs for overview, expandable to deeper content
