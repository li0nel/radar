Orchestration
Agent Teams' peer-to-peer messaging enables production coordination patterns

The critical distinction between subagents and Agent Teams is not the number of agents. It's the communication topology. Subagents use fan-out/fan-in: a parent agent spawns children, each child works independently, each child reports results back to the parent, and the parent synthesizes. The parent is a serialization bottleneck. If the frontend agent discovers that the API contract needs to change, it cannot tell the backend agent directly—it must report back to the parent, which then relays the message to the backend agent. In a fan-out/fan-in model, every cross-agent communication has double the latency because it routes through the hub.

Agent Teams use a lead-mate architecture with peer-to-peer messaging. The lead agent creates mates and assigns initial tasks, but mates can message each other directly using sendMessage. When the frontend mate discovers the API contract needs adjustment, it messages the backend mate directly. No round-trip through the lead. The lead is still the orchestrator—it created the mates, assigned domains, and will synthesize the final result—but it is not a mandatory relay for all inter-agent communication. This is the architectural difference that matters for production coordination.

Configuration lives in .claude/teams.yml at the project root. The file declares team members, their roles, their domain boundaries, and their communication permissions. A typical full-stack configuration:

team:
  name: feature-auth-oauth
  lead:
    role: architect
    context: "Overall feature design and integration"
  mates:
    backend:
      role: "Backend engineer"
      domain: "src/api/**, src/services/**, src/models/**"
      context: "Express.js API, PostgreSQL, OAuth2 flows"
      can_message: [frontend, tests]
    frontend:
      role: "Frontend engineer"
      domain: "src/components/**, src/hooks/**, src/pages/**"
      context: "React 19, TypeScript, TanStack Query"
      can_message: [backend, tests]
    tests:
      role: "Test engineer"
      domain: "tests/**, src/**/*.test.ts"
      context: "Vitest, Testing Library, MSW for mocking"
      can_message: [backend, frontend]
  worktree:
    isolation: true
    base_branch: main

The domain field is enforced, not advisory. A mate can only edit files matching its domain globs. This is the single most important configuration decision in the entire file. Get the domain boundaries wrong and you get merge conflicts. Get them right and mates can work in parallel without stepping on each other. The rule is simple: no file should match more than one mate's domain. If src/types/api.ts is needed by both backend and frontend, it must be assigned to exactly one mate (typically the one that defines the types) and the other mate reads it but cannot modify it.

The worktree isolation model deserves detailed explanation because it's the mechanism that makes parallel work safe. When isolation: true is set, each mate gets its own git worktree branched from the base. The backend mate works in .worktrees/feature-auth-oauth/backend/, the frontend mate in .worktrees/feature-auth-oauth/frontend/, and so on. Each worktree is a full copy of the repository at the branch point, with its own working directory and index. Mates commit to their own branches. When all mates complete, the lead merges the branches.

This is where most teams encounter their first real failure mode. The merge step is not trivial. If the backend mate changed a shared type definition and the frontend mate was consuming the old version of that type, the merge succeeds (no textual conflict) but the build fails (type mismatch). Worktree isolation prevents git conflicts but does not prevent semantic conflicts. The peer-to-peer messaging exists precisely to prevent this: when the backend mate changes a shared interface, it should message the frontend mate with the new contract before proceeding.

The practical communication pattern that works is what I'll call notify-on-contract-change. Each mate is responsible for messaging any mate that consumes its outputs whenever those outputs change shape. This is a convention, not something Agent Teams enforces automatically. You encode it in the mate's context field:

mates:
  backend:
    role: "Backend engineer"
    domain: "src/api/**, src/services/**"
    context: |
      Express.js API, PostgreSQL, OAuth2 flows.
      CRITICAL: When you change any API endpoint signature,
      response shape, or error format, immediately message
      the frontend mate with the new contract before
      continuing your work.

When to use subagents vs. Agent Teams is a decision tree, not a preference. Subagents are correct when: the tasks are genuinely independent (no shared artifacts), each task is small enough to fit in a single agent's context, results don't need to influence each other during execution, and the parent can synthesize results without cross-referencing. Example: "Write unit tests for these five independent utility functions." Five subagents, five test files, no interaction needed. Fan-out, fan-in, done.

Agent Teams are correct when: tasks span multiple domains that share artifacts (API types, database schemas, configuration files), intermediate results from one domain need to influence work in another domain during execution (not just at the end), and the work is large enough that a single agent's context window cannot hold the full picture. Example: "Implement OAuth login across the API, the React frontend, and the test suite." The API mate defines the endpoints, the frontend mate consumes them, the test mate mocks them—they need to agree on contracts in real time.

The wrong choice in each direction has characteristic failure modes. Using subagents when you need Agent Teams produces integration failures at merge time—each subagent made locally-correct decisions that are globally inconsistent. Using Agent Teams when subagents suffice wastes tokens on coordination overhead and introduces unnecessary complexity—each sendMessage consumes tokens in both the sender's and receiver's context windows.

Domain-split strategies determine the success or failure of the entire team. The most common mistake is splitting by layer (one mate for all models, one for all controllers, one for all views) instead of by feature slice. Layer splits guarantee cross-mate dependencies because every feature touches every layer. Feature splits minimize cross-mate communication because each mate owns a vertical slice. The exception is when you're doing a pure refactoring across a single layer—then a layer split makes sense because the work is genuinely confined to one layer.

For a typical feature implementation, the recommended split is: one mate per bounded context (backend API for auth, frontend UI for auth, shared types/config), plus one mate for tests. The test mate is almost always worth having as a separate entity because test writing is embarrassingly parallel and tests provide the verification layer for the other mates' work. The test mate receives contracts from the other mates via messages and writes tests against those contracts, catching integration mismatches before the final merge.

Error recovery in Agent Teams has subtleties that the documentation glosses over. When a mate fails (crashes, exceeds context, or produces output that fails validation), the lead has three options: restart the mate with additional context about what went wrong, reassign the mate's remaining work to another mate or to itself, or abort the entire team and fall back to sequential single-agent work. The choice depends on the failure mode. A crash from context exhaustion means the task was too large for one mate—splitting the domain further or summarizing intermediate state and restarting is the right response. A semantic failure (mate produced code that doesn't compile or violates contracts) means the mate's context or domain boundaries were wrong—adjusting the teams.yml and restarting is more productive than retrying with the same configuration.

The lead agent's context window is the hidden constraint. Every message between mates is also visible to the lead (for coordination purposes), which means the lead's context fills up faster than any individual mate's. For large teams (4+ mates), the lead can hit context limits before any mate does. The mitigation is to keep the lead's role strictly to orchestration—it should not be doing implementation work. If the lead is also writing code, it's combining orchestration overhead with implementation overhead in the same context window, and it will exhaust context first.

A production pattern that has emerged from teams using Agent Teams at scale is the checkpoint-merge cycle. Instead of having all mates work to completion and then merging once at the end, the lead periodically (every 30-60 minutes of wall time, or after each major subtask) pauses all mates, merges their current branches into a shared integration branch, runs the build and test suite against the integrated code, and then has each mate pull from the integration branch before continuing. This catches integration failures incrementally instead of discovering them all at the end. It's more expensive (more merges, more build runs) but dramatically reduces the risk of a catastrophic merge failure after hours of parallel work.

The TeammateIdle hook event fires when a mate completes its assigned work and is waiting for new tasks. This is useful for dynamic work rebalancing: if the backend mate finishes early, the lead can assign it documentation or code review tasks for the frontend mate's output. The TaskCompleted event fires when a specific task (not the entire mate's work) completes, enabling finer-grained orchestration. Both events are recent additions (the 17 lifecycle events documented elsewhere) and are not yet widely used in production configurations.

One non-obvious constraint: Agent Teams requires Claude Code on all mates. You cannot mix vendors. If your backend is best served by Codex and your frontend by Claude, Agent Teams cannot help—you need ACPX for cross-vendor orchestration. Agent Teams is Anthropic's first-party multi-agent system and it only speaks Claude. This is the primary reason both Agent Teams and ACPX exist in the ecosystem: they solve different slices of the multi-agent problem.

Start with a two-mate team (implementation + tests) before attempting three or more. The domain boundary definition is the hardest part and the one most likely to cause failures. Validate with one feature, then expand. The peer-to-peer messaging for contract changes is the pattern that differentiates Agent Teams from just running multiple subagents.
