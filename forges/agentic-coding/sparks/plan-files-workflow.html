WORKFLOW DESIGN
Plan files in the repo beat tickets for agentic development

George Violaris published the most complete description of an agentic development workflow in early 2026, and the centerpiece is not a tool or a prompt technique—it is a file format. A plan file. One per feature, living in the repository, version-controlled alongside the code it describes. The plan file is the contract between the human (who defines what to build) and the agent (who builds it). Everything else in the workflow exists to produce, refine, or execute plan files.

The workflow has four phases. Phase 1 is brainstorm: you describe the feature you want, and the agent maps it against the actual codebase context. Not in the abstract—against the real files, the real APIs, the real data models. The output of Phase 1 is a feature map: what exists that is relevant, what needs to change, what needs to be created, what constraints the codebase imposes. This phase is exploratory. You are not committing to anything. You are building shared understanding between yourself and the agent about the shape of the work.

Phase 2 converts the feature map into plan files. This is where the real work happens, and Violaris is explicit about this: "Implementation is the easy part. The plan is where the value is." Each plan file has five sections, and every section is mandatory:

# Plan: [Feature Name]

## Objective
[Observable behavior, not implementation details.
"Users can reset their password via email" not
"Add a POST /reset-password endpoint"]

## Constraints
[What this feature must NOT do. Boundaries.
"Must not require re-authentication after reset"
"Must not send more than one email per request"
"Must not modify the existing session table schema"]

## Dependencies
[Other features, services, or files this depends on.
"Requires: email service (already implemented in src/email/)
Requires: user table has email column (confirmed)
Blocked by: nothing"]

## Acceptance Criteria
[Specific, testable conditions. Each one is a yes/no.
"[ ] User receives reset email within 30 seconds
 [ ] Reset link expires after 24 hours
 [ ] Used reset link returns 410 Gone
 [ ] Password must meet existing strength requirements
 [ ] Audit log entry created for every reset attempt"]

## Open Questions
[Things you do not know yet. Unresolved decisions.
"Should we rate-limit reset requests per email address?
Do we need to invalidate existing sessions on reset?
What is the email template? (Waiting on design)"]

The Objective section is the most important and the most commonly written badly. Violaris insists on observable behavior: what the user sees, not what the code does. "Add a password reset endpoint" is an implementation directive that constrains the agent's solution space unnecessarily. "Users can reset their password via email" describes the outcome and lets the agent figure out the implementation. The distinction matters because agents are better at implementation when you tell them what to achieve, and worse when you tell them how to achieve it.

The Constraints section is the section most people skip, and its absence causes the most rework. Without explicit constraints, the agent makes reasonable but wrong assumptions. It refactors the session table to add a reset_token column when you wanted a separate tokens table. It adds a new email service when you wanted it to use the existing one. It requires re-authentication after reset when the product requirement is seamless continuation. Every constraint you fail to write is a 50/50 chance the agent guesses wrong, and each wrong guess is 15–45 minutes of rework.

The Dependencies section prevents the most expensive failure: building on something that does not exist or is not ready. The agent starts implementing a feature that depends on an API endpoint that was planned but never built. Thirty minutes of work wasted. Explicit dependencies force you to verify that prerequisites are met before starting, and they give the agent the exact file paths and interfaces it needs to integrate with.

Acceptance Criteria must be specific and testable. "Works correctly" is not an acceptance criterion. "Reset link expires after 24 hours" is. Each criterion maps directly to either an automated test (Gate 1) or a verification step in the Stop hook. When the agent says it is done, the Stop hook checks the plan file's acceptance criteria and verifies each one. This is the closed loop: the plan file defines done, the implementation attempts done, the Stop hook verifies done.

Open Questions is the section that makes plan files honest. Every feature has unknowns. Pretending otherwise leads to agents making silent assumptions about the unknowns, which leads to rework when the assumptions are wrong. By listing open questions explicitly, you create a forcing function: resolve the questions before starting implementation, or accept that the implementation may need to change when the questions are answered. The plan file makes the uncertainty visible instead of hiding it in the agent's implicit decisions.

Phase 3 is implementation. Violaris describes it as "the easy part," and this is not false modesty—it is a structural observation. With a well-written plan file, implementation is mechanical. The agent knows what to build (objective), what not to do (constraints), what already exists (dependencies), and what "done" means (acceptance criteria). The degrees of freedom are narrow enough that the agent can make good decisions within the defined space. The quality of the implementation is directly proportional to the quality of the plan.

Phase 4 is review. The plan file serves as the review checklist. The reviewer (human, Gate 3) reads the plan file and the implementation together. Does the code achieve the objective? Does it violate any constraints? Does it correctly integrate with the dependencies? Does it satisfy all acceptance criteria? Were the open questions resolved, and if so, are the resolutions reflected in the code? The plan file transforms code review from "does this code look right?" to "does this code match the contract?"

Violaris's key insight deserves direct quotation: "The value of agentic coding isn't speed. It's that it forces you to be explicit about what you'd normally leave implicit." In traditional development, a senior engineer carries implicit knowledge about constraints, dependencies, and acceptance criteria. They do not write them down because they hold them in their head and apply them continuously during implementation. An agent has no implicit knowledge. Everything must be explicit or it does not exist. The plan file is the mechanism for making implicit knowledge explicit, and the surprising side effect is that the explicit version is better—more complete, more precise, more reviewable—than the implicit version ever was.

Why files beat tickets. The comparison to Jira tickets, Linear issues, or GitHub issues is inevitable, and the answer is structural, not preferential. Plan files are versioned. When the plan changes, git diff shows exactly what changed and when. Tickets have edit histories, but they are append-only and hard to diff. Plan files are diffable. You can compare v1 of the plan to v3 and see the evolution of requirements. Tickets accumulate comments but the canonical state is ambiguous. Plan files survive session resets. When an agent session ends and a new one begins, the plan file is right there in the repo. No need to paste a ticket URL, no need to re-explain context. The new session reads the plan file and has full context. Plan files live next to the code. The plan for the password reset feature lives in plans/password-reset.md right next to src/auth/reset.ts. Proximity matters for both humans and agents.

Plan files also compose. A large feature decomposes into multiple plan files, each covering a sub-feature. The dependencies section creates a DAG (directed acyclic graph) of plan files, and the implementation order falls out naturally: implement the plans with no unresolved dependencies first, then the plans that depend on them, and so on. This is project management via filesystem and git, not via a separate tool with its own state management and synchronization problems.

Integration with voice-first workflows. For developers using voice transcription (Wispr Flow, Whisper, or similar), the plan file workflow has a natural on-ramp. Start with a 10–20 minute voice dump: describe the feature, what it should do, what it should not do, what you are unsure about, what it depends on. Dump the raw transcription into the agent and instruct it to produce a plan file with the five sections. The agent converts your stream-of-consciousness into a structured contract. You review the plan file, correct misunderstandings, add missing constraints, and resolve what you can of the open questions. Iterate until the plan file is airtight. Then implement.

This voice-to-plan pipeline solves the "blank page" problem that many developers face when asked to write specs. Speaking is easier than writing for most people, especially when the topic is a feature they understand well but have not yet structured. The agent's role in Phase 2 is to provide the structure, not the content. You provide the knowledge; the agent provides the format.

The plan file template should live in your repository as a template that agents can reference. Place it at .claude/templates/plan-file.md or equivalent, and reference it in CLAUDE.md so every session knows the format exists. The agent should never have to invent the plan file format from scratch—it should copy the template and fill in the sections.

One operational refinement: plan files should have a status header. Status: draft means the plan is being written and is not ready for implementation. Status: approved means the plan has been reviewed and implementation can begin. Status: in-progress means implementation has started. Status: complete means all acceptance criteria are met and the plan is done. This status field serves as a simple state machine that both humans and agents can check before acting. An agent should refuse to implement a draft plan and should refuse to modify a complete plan without explicit instruction.

The plan file workflow is not faster than "just start coding." It is slower in Phase 2 (writing plans takes 20–40 minutes per feature) and dramatically faster in Phase 3 (implementation with a good plan is 2–5x faster than implementation without one) and Phase 4 (review against a plan is 3–10x faster than review without one). The total time is comparable or slightly less, but the defect rate drops significantly because the most common source of defects—misunderstood requirements—is addressed before a single line of code is written.

The plan file workflow is the single most effective process change for agentic development. Start with one feature. Write the plan file. Implement against it. Measure the difference in rework. The upfront time investment in Phase 2 pays for itself in Phase 3 and 4 on the first feature.
