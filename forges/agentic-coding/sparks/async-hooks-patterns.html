Hooks & Configuration
Async hooks enable non-blocking side effects without slowing the agent loop

Every synchronous hook on PostToolUse is a tax on every tool invocation. If you have a formatter that runs on file writes and it takes 150ms, and Claude writes 40 files during a feature implementation, you've added 6 seconds of dead time. Not model thinking time, not network latency—just your hook blocking the agent loop while it does something Claude doesn't need to see the result of. Released in January 2026, the "async": true flag on hook definitions solves this by running the hook in the background with fire-and-forget semantics. The hook launches, Claude continues immediately, and the hook's output is discarded.

The syntax is one line added to any existing hook definition:

{
  "hooks": {
    "PostToolUse": [
      {
        "type": "command",
        "command": "bash ~/.claude/hooks/format-on-save.sh",
        "async": true
      }
    ]
  }
}

That's it. The hook still receives the same event JSON on stdin. It still runs the same command. But Claude Code spawns the process and does not wait for it to exit. No return value is captured, no exit code is checked, no stdout is read. The hook runs in its own process group and must manage its own error handling—if it fails, nobody notices unless the hook itself writes to a log file or sends an alert.

The design decision to fully discard async hook output is intentional and has a specific implication: async hooks cannot influence Claude's behavior. A synchronous PreToolUse hook can return exit code 2 to block a tool invocation. An async PreToolUse hook cannot—by the time it decides to block, Claude has already moved on. This is not a bug; it's the core tradeoff. Async means "I don't care about the result" and that necessarily means "I cannot influence the outcome."

This creates a clean decision framework for every hook you write. Ask one question: does Claude need to see this hook's result before proceeding? If yes, the hook must be synchronous. If no, make it async. The framework maps cleanly onto hook categories:

Must be synchronous: Security blocking via PreToolUse (exit 2 to deny dangerous commands). Auto-approve decisions via PermissionRequest (the hook is the decision). Any hook that returns systemMessage or additionalContext that Claude needs to read. Skill activation on UserPromptSubmit (Claude needs the injected skill context before processing the prompt). Model routing on UserPromptSubmit (the model selection must happen before inference starts).

Should be async: Logging and analytics (PostToolUse telemetry to a centralized service). Backup creation (PreCompact state snapshots). Desktop notifications (Stop, Notification events). Metrics collection (session duration, tool invocation counts). File watchers (triggering external build systems after writes). Slack/webhook delivery. Any side effect where the only consumer is a human or an external system, not Claude itself.

The PreCompact backup pattern is the canonical example of async hooks done right. Context compaction is a critical moment in a Claude Code session—the model is about to lose detailed memory of earlier conversation turns. A PreCompact hook can capture the full conversation state before compaction occurs, writing it to a backup file or sending it to a persistence service. This backup must not delay the compaction itself, because compaction typically triggers when the context window is nearly full and Claude needs the space to continue working. If the backup hook takes 2 seconds synchronously, that's 2 seconds of blocked agent operation at exactly the moment when the system is under memory pressure. Making the backup async means the compaction proceeds immediately while the backup writes to disk in parallel.

{
  "hooks": {
    "PreCompact": [
      {
        "type": "command",
        "command": "bash ~/.claude/hooks/backup-context.sh",
        "async": true
      }
    ]
  }
}

The backup script receives the pre-compaction state on stdin, writes it to a timestamped file in .claude/backups/, and optionally syncs to remote storage. Because it's async, a slow disk write or a network timeout for the remote sync doesn't block compaction. The tradeoff is that if the backup script crashes, the state is lost and nobody is notified inline. The script needs its own error handling—write to a local error log, send an alert, retry with exponential backoff. This is the operational overhead of async hooks: you trade latency for observability, and you must build the observability back yourself.

The performance impact of sync hooks on PostToolUse deserves specific measurement because this is the highest-frequency lifecycle event. In an active coding session, Claude might invoke tools 100-300 times. Every sync hook on PostToolUse runs on every one of those invocations. Common PostToolUse hooks include code formatters (prettier, black, rustfmt), linters (eslint, pylint), type checkers (tsc --noEmit), and telemetry loggers. A formatter that takes 100ms per invocation adds 10-30 seconds to a session. A type checker that takes 500ms adds 50-150 seconds. These numbers compound: two sync hooks at 200ms each on 200 tool invocations is 80 seconds of dead time.

The fix is straightforward: make everything that isn't a blocking decision async. The formatter? If it's for human readability and Claude doesn't read the formatted output, make it async. The linter? If it's informational and Claude doesn't adjust behavior based on lint results, make it async. The type checker? This one is nuanced—if Claude uses type errors to guide its next edit, it must be sync. If you're just collecting type error counts for a dashboard, async. The telemetry logger? Always async. There is no scenario where Claude needs to wait for a telemetry write to complete before proceeding.

Desktop notifications are the simplest async hook use case. On macOS:

#!/bin/bash
# notify-on-stop.sh - async hook for Stop event
EVENT=$(cat -)
STATUS=$(echo "$EVENT" | jq -r '.status // "completed"')
osascript -e "display notification \"Claude Code session ${STATUS}\" with title \"Claude Code\""

On Linux with notify-send, or on any platform via a webhook to your phone's notification service. The hook fires when Claude finishes a task (the Stop event), and the notification arrives a few hundred milliseconds later. If the notification fails—maybe the notification daemon isn't running—it doesn't matter. Claude already stopped. The notification is purely informational.

There's a subtlety with async hooks and process lifecycle. When Claude Code exits, it does not wait for async hooks to complete. If an async Stop hook is writing a summary to disk and Claude Code's process terminates, the hook's process may be killed mid-write (depending on the OS and process group handling). For hooks that perform important writes on session-ending events (Stop, SessionEnd), you need to handle this. The standard approach is to write to a temporary file first, then atomically rename it to the final path. If the process is killed during the write, you lose only the temp file. If it completes, the rename is atomic and the data is safe.

#!/bin/bash
# safe-async-write.sh
TMPFILE=$(mktemp)
cat - > "$TMPFILE"
FINAL="$HOME/.claude/session-logs/$(date +%s).json"
mv "$TMPFILE" "$FINAL"  # atomic on same filesystem

Another pattern worth documenting: the async audit trail. A sync PostToolUse hook that logs every tool invocation to a centralized audit service adds network latency to every tool call. If the audit service is behind a load balancer with a 50ms P50 and a 500ms P99, you're adding at least 50ms per tool invocation and occasionally half a second. Over 200 invocations, the P50 overhead is 10 seconds and the P99 scenarios could add over a minute. Making this hook async eliminates the latency entirely. The audit records arrive at the service slightly out of order (because async hooks don't guarantee ordering), but for audit purposes, each record includes a timestamp, so the service can reconstruct the timeline.

The ordering guarantee (or lack thereof) is important. Sync hooks execute in the order they're defined in the hooks array and each completes before the next starts. Async hooks all launch simultaneously and complete in arbitrary order. If you have two async PostToolUse hooks—one that writes to a local log and one that sends to a remote service—there's no guarantee about which writes first. For independent side effects this doesn't matter. For dependent side effects (where hook B reads what hook A wrote), you must use sync hooks or implement your own coordination mechanism.

How to audit your existing hooks for async candidates: list every hook in your configuration. For each one, answer three questions. First, does Claude read this hook's output? If the hook returns systemMessage, additionalContext, or influences tool approval via exit codes, it must be sync. Second, does the hook's timing matter? If the hook must complete before the next tool invocation starts (e.g., a lock file that prevents concurrent edits), it must be sync. Third, who is the audience for this hook's output? If the audience is Claude, sync. If the audience is a human, a dashboard, a logging service, or an external system, async.

In practice, most hooks in a mature configuration should be async. A typical setup might have 8-12 hooks across various lifecycle events. Of those, usually 2-3 need to be sync (security blocking, permission handling, skill activation) and the rest are side effects that should be async. The performance difference between a mostly-sync and a mostly-async hook configuration is measurable in wall-clock seconds per session. On a 30-minute coding session with 200+ tool invocations, converting five 100ms sync hooks to async saves 100 seconds. That's not theoretical—that's time you sit watching Claude's spinner instead of reading Claude's output.

One final pattern: the hybrid approach. For hooks where you want async performance but occasional sync behavior, use a conditional wrapper. The hook checks a fast condition (does a file exist? is an environment variable set?) synchronously in under 1ms. If the condition is met, it does synchronous work. If not, it spawns a background process and exits immediately, mimicking async behavior within a sync hook definition. This is a workaround for the fact that the async flag is per-hook-definition, not per-invocation. You can't say "be async usually, but sync when it matters." The conditional wrapper achieves the same effect at the script level.

#!/bin/bash
# hybrid-formatter.sh - sync hook that acts async when safe
EVENT=$(cat -)
TOOL=$(echo "$EVENT" | jq -r '.tool')

if [ "$TOOL" = "Write" ]; then
  FILE=$(echo "$EVENT" | jq -r '.input.file_path')
  # Only format synchronously if Claude will read the file next
  if echo "$EVENT" | jq -e '.context.next_tool == "Read"' > /dev/null 2>&1; then
    prettier --write "$FILE" 2>/dev/null
  else
    # Fire-and-forget background formatting
    nohup prettier --write "$FILE" > /dev/null 2>&1 &
  fi
fi

Audit every PostToolUse hook in your configuration right now. If the hook is logging, formatting, or notifying—and Claude never reads its output—add "async": true. The cumulative latency savings across a day of agentic coding sessions are significant enough to justify the five minutes of configuration changes.
