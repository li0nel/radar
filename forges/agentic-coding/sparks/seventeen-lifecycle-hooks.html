Hooks & Configuration
Claude Code exposes 17 lifecycle events — the newer ones unlock Agent Teams and automated onboarding

The hook system started with five events: PreToolUse, PostToolUse, Notification, Stop, and UserPromptSubmit. That covered the basics—intercept before a tool runs, react after it runs, catch notifications, handle session end, process incoming prompts. Over the past four months, the event surface has expanded to 17 events, and the newer additions reveal where Claude Code is heading architecturally. The full list as of March 2026: SessionStart, UserPromptSubmit, PreToolUse, PermissionRequest, PostToolUse, PostToolUseFailure, SubagentStart, SubagentStop, Stop, PreCompact, Setup, SessionEnd, Notification, TeammateIdle, TaskCompleted, ConfigChange, WorktreeCreate, and WorktreeRemove.

These fall into four generations. The original five (PreToolUse, PostToolUse, Notification, Stop, UserPromptSubmit) are the tool-lifecycle hooks—they let you intercept and react to Claude's tool usage. The session-lifecycle additions (SessionStart, SessionEnd, PreCompact, PermissionRequest, PostToolUseFailure, Setup) expanded coverage to session boundaries, context management, and error handling. The sub-agent hooks (SubagentStart, SubagentStop) arrived with Claude Code's internal sub-agent architecture. And the newest five (TeammateIdle, TaskCompleted, ConfigChange, WorktreeCreate, WorktreeRemove) are the Agent Teams hooks, designed for multi-instance coordination.

The five Agent Teams events deserve detailed attention because they're the least documented and the most architecturally significant.

TeammateIdle fires when a teammate instance in an Agent Teams configuration has no assigned work. The event payload includes the teammate's ID, its current worktree, the last task it completed, and its idle duration. This is the scheduling hook—it's how you implement work-stealing, load balancing, and dynamic task assignment. Without this event, an idle teammate just sits there consuming resources until a human notices and assigns it something. With a TeammateIdle hook, you can automatically pull the next task from a queue, assign review work for code another teammate just wrote, or trigger the teammate to run tests on recent changes. The hook response can include a task assignment that the idle teammate picks up immediately.

{
  "event": "TeammateIdle",
  "teammate": {
    "id": "teammate-3",
    "worktree": "/project/.worktrees/feature-auth",
    "lastTask": "Implement OAuth token refresh",
    "idleSince": "2026-03-31T14:22:00Z",
    "idleDuration": 45
  }
}

A practical TeammateIdle hook checks a task queue (a JSON file, a Redis list, a GitHub Projects board via API) and returns a task assignment. If the queue is empty, it can assign housekeeping work: run the test suite, check for type errors, lint recently changed files, update documentation for files modified in the last hour. The teammate stays productive instead of idle.

TaskCompleted fires when any teammate finishes a task. The payload includes the teammate ID, the task description, the outcome (success, failure, partial), a summary of changes made, and the list of files modified. This is the coordination event—it lets other teammates react to completed work. A TaskCompleted hook can trigger code review by another teammate, run integration tests that depend on the completed work, update a shared progress tracker, or notify downstream teammates that a dependency is resolved. In a multi-teammate setup where teammate A implements a module and teammate B writes tests for it, a TaskCompleted hook on A's completion can automatically assign the testing task to the next idle teammate.

ConfigChange fires when Claude Code's settings are modified during a session. This includes changes to permissions, model selection, MCP server configuration, and hook definitions. The payload contains the setting path, old value, new value, and source (user action, managed policy update, or programmatic change). The primary use case is audit logging—tracking who changed what and when in a team environment. A secondary use case is reactive configuration: if someone disables a security-critical hook, a ConfigChange hook (defined at a higher priority level, like managed policy) can detect this and re-enable it, or alert an admin, or force-terminate the session. This is the self-healing configuration pattern—the configuration monitors itself.

WorktreeCreate and WorktreeRemove fire when Agent Teams creates or removes git worktrees for teammate instances. Each teammate in an Agent Teams setup operates in its own worktree to prevent file conflicts—teammate A editing src/auth.ts in one worktree while teammate B edits src/db.ts in another. The WorktreeCreate hook fires after the worktree is set up and before the teammate starts working. This is the initialization hook for per-teammate environments: install dependencies in the new worktree, set up local configuration, warm caches, verify the build system works. WorktreeRemove fires before cleanup, giving you a chance to archive logs, capture metrics, or backup any unsaved state from the worktree.

# WorktreeCreate hook - initialize new teammate worktree
#!/bin/bash
EVENT=$(cat -)
WORKTREE=$(echo "$EVENT" | jq -r '.worktree.path')
TEAMMATE=$(echo "$EVENT" | jq -r '.worktree.teammate')

# Install dependencies in the new worktree
cd "$WORKTREE" && npm install --silent 2>/dev/null

# Copy local env config
cp ~/.claude/team-env/.env.local "$WORKTREE/.env.local" 2>/dev/null

# Log
echo "$(date) Initialized worktree for $TEAMMATE at $WORKTREE" \
  >> ~/.claude/hooks/worktree.log

The Setup event is separate from the Agent Teams hooks but equally important for team-scale operations. It triggers when a developer runs claude --init or claude --maintenance. This is the automated onboarding hook—when a new developer joins the team and initializes Claude Code for the first time, the Setup hook runs and performs all the first-time configuration that would otherwise be a checklist in a wiki: install project dependencies, verify required tools are present (correct Node version, required CLI tools, docker running), configure MCP servers for the project, copy team-standard hook scripts into place, validate API key permissions, and generate a local configuration file from a template.

The power move with Setup hooks is combining them with Claude's agentic capabilities. The command claude --init "/install" triggers two things in sequence: first, the Setup hook runs (deterministic, infrastructure-level), and then the /install command executes (agentic, LLM-driven). The hook handles the mechanical setup that must be correct—installing the right dependency versions, configuring environment variables, setting file permissions. The /install agentic phase handles the judgment-requiring setup—analyzing the codebase to determine what MCP servers would be useful, reading the README to understand build procedures, identifying project-specific configuration that the hook template doesn't cover. Deterministic first, agentic second. This ordering is deliberate: the agentic phase can assume a correctly-configured environment because the deterministic phase already ensured it.

The four handler types available for all 17 events form a capability hierarchy. Command handlers run a shell command, receive the event on stdin, and communicate via exit codes and stdout. Fast, simple, limited to what shell scripts can do. HTTP handlers POST the event JSON to a URL and interpret the response. Good for centralized services, constrained by network latency. Prompt handlers send the event to an LLM for evaluation—the LLM reads the event payload and returns a judgment. This is useful for nuanced decisions that can't be expressed as regex or conditionals, but adds LLM latency and cost to every event. Agent handlers are the most powerful: they spawn a sub-agent with full tool access (Read, Grep, Glob, Bash) that can examine the codebase before making a decision.

Agent-type handlers are transformative for Stop and SubagentStop events. A command-type Stop hook can check whether the last tool invocation succeeded. An agent-type Stop hook can read the files that were modified, grep for common error patterns, run the test suite, check for uncommitted changes, verify that the code compiles, and then decide whether to allow the session to end or inject a message telling Claude to keep working. This turns the Stop event from a simple checkpoint into a quality gate. The agent examines the work product and decides whether it meets the bar.

{
  "hooks": {
    "Stop": [
      {
        "type": "agent",
        "prompt": "Review the changes made in this session. Check that all modified files compile, that no TODO comments were left unresolved, that test files exist for new modules, and that no debugging code (console.log, debugger statements) remains. If any issues are found, return a message asking Claude to fix them before stopping.",
        "tools": ["Read", "Grep", "Glob", "Bash"]
      }
    ]
  }
}

The cost of agent-type handlers is real: each invocation is a full LLM inference with tool use, potentially taking 10-30 seconds and consuming thousands of tokens. Use them only on low-frequency events (Stop, SubagentStop, SessionEnd) where the quality gate justifies the latency. On high-frequency events like PostToolUse, an agent handler would make Claude unusably slow.

Configuration priority determines which hooks actually fire when multiple sources define hooks for the same event. The hierarchy, from highest to lowest priority: managed policy (set by organization admins, cannot be overridden), project shared (.claude/settings.json in the repo, visible to all developers), project personal (.claude/settings.local.json, gitignored, per-developer), and user global (~/.claude/settings.json, applies to all projects). Hooks at all levels fire—this is additive, not overriding. A managed policy hook on PreToolUse runs alongside a project-level hook on PreToolUse. But if they conflict (one allows, one blocks), the higher-priority source wins. Managed policy blocking cannot be overridden by any lower-priority hook.

This priority system is what makes Claude Code viable for enterprise use. The security team defines managed policy hooks that enforce organizational requirements: no credentials in code, no force-push to main, no execution of unvetted scripts, audit logging of all tool invocations. Individual teams add project-level hooks for their specific workflows: testing conventions, formatting rules, documentation requirements. Individual developers add personal hooks for their preferences: desktop notifications, custom keybindings, experimental features. Each layer adds value without being able to undermine the layers above it.

The 17 events together form a complete observability and control surface for the agentic coding lifecycle. From session initialization through prompt processing, tool execution, permission management, sub-agent coordination, context compaction, teammate scheduling, worktree management, configuration changes, and session termination—every phase is hookable. The question is no longer "can I hook into this?" but "what should I do with the hook?" The answer depends on your scale: solo developers need 2-3 hooks (PreToolUse for safety, Stop for verification, PreCompact for backup). Teams need 5-8 (add PostToolUse formatting, UserPromptSubmit skill activation, Setup for onboarding). Organizations running Agent Teams need 10+ (add the teammate coordination events, ConfigChange auditing, WorktreeCreate initialization).

Start with SessionStart, PreToolUse, Stop, and PreCompact as your base four hooks. These cover initialization, safety, completion verification, and context preservation. Add the Agent Teams events only when you're actually running multi-teammate workflows—hooking events that never fire is harmless but creates configuration noise. The Setup hook is the highest-ROI addition for any team with more than three developers; it eliminates onboarding friction completely.
