Orchestration
ACP protocol and ACPX replace PTY scraping for cross-vendor agent orchestration

The Agent Client Protocol (ACP), created by Zed Industries and now co-maintained by JetBrains, solves the M×N integration problem for coding agents the same way LSP solved it for language tooling. Instead of spawning child processes and scraping raw terminal output—losing structured metadata, diffs, thinking steps, and error semantics—ACP uses JSON-RPC 2 over stdio to exchange typed events between orchestrator and agent.

ACPX (openclaw/acpx, MIT) is the headless CLI client. It gives you named persistent sessions, prompt queueing, crash reconnect, and fire-and-forget mode. From Claude Code, you can delegate to Codex, Gemini, Kiro, or any of 15+ supported agents through structured protocol communication. Results come back as typed events you can pipe through jq, not mangled ANSI streams. Anthropic declined native ACP support (issue #6686, 439 thumbs-up), so ACPX bridges the gap with a CLI adapter.

You're single-agent today. ACPX lets you delegate tests to Codex while Claude implements, or have Gemini review Claude's output—all structured, no scraping. Pair with cmux for the full multi-agent terminal setup.

Agent Teams' peer-to-peer messaging is the real differentiator from subagents

Claude Code's Agent Teams let multiple Claude Code sessions collaborate as a coordinated team. The architectural distinction from subagents is peer-to-peer communication: subagents report results back to a parent (fan-out/fan-in), but Agent Teams teammates message each other directly, share discoveries mid-task, and pass intermediate artifacts without the lead acting as intermediary. The lead-mate architecture uses .claude/teams.yml for team structure, with the lead handling decomposition and mates owning domains.

Use subagents for independent parallel tasks (tests, linting, type-checking) where the parent only needs aggregated results. Use Agent Teams when developing across domains simultaneously (frontend + backend + infra), when intermediate artifacts like API schemas need real-time sharing, or when you want agents to review each other's work.

Your current subagent usage caps out at fan-out/fan-in. For large full-stack features—your stated pain point—Agent Teams let a backend mate share a schema with a frontend mate directly, eliminating the serialization bottleneck that causes regressions.

Orchestration
The conductor-to-orchestrator transition demands decomposition skills, not coding skills

Addy Osmani's "Code Agent Orchestra" talk at O'Reilly AI CodeCon codifies an 8-level maturity model for AI-assisted coding. Most developers are stuck at levels 3–4 (interactive pair programming). The orchestration tier starts at level 6 and requires a fundamentally different skillset: clear specs, work decomposition, and output verification rather than writing code yourself. The mental model shifts from pair programming to managing a team.

Three walls force the transition: context overload (one agent can't hold a large codebase), no specialization (a generalist writes worse database code than a focused agent), and no coordination (sequential work when parallel is possible). The practical discipline: plan the work, assign it, verify the output. Your ceiling is no longer what fits in one context window.

This maps directly to your pain point with big full-stack features ending in waste. The fix isn't better prompting—it's better decomposition. Osmani's framework gives you the maturity ladder to self-assess where you are (likely level 5) and what level 6–7 looks like.

cmux is the purpose-built terminal for multi-agent workflows on macOS

The workflow: Claude Code orchestrates in one pane, ACPX sessions run in others, cmux lights up completed panes and fires desktop notifications. Running three agents in three terminal tabs is chaos; cmux's notification system turns it into something manageable.

If you move to multi-agent (Agent Teams or ACPX), cmux replaces your terminal multiplexer with something purpose-built. The notification system means you can context-switch to other work while agents run, then come back when they need you.

Orchestration
Microsoft's Swarm Diaries: five agents on one codebase exposed the coordination failure modes

Microsoft's Azure Apps team ran five AI agents simultaneously on a single codebase and published "The Swarm Diaries" documenting exactly what broke. The primary failure modes: merge conflicts from parallel edits to shared files, inconsistent architectural decisions when agents lacked shared context, and cascading test failures when one agent's changes broke another's assumptions. The wins: 3–5x throughput on truly independent workstreams with clear file boundaries.

Their key finding: multi-agent only works when work is decomposed into genuinely independent units with explicit interface contracts. Shared files need a single owner. The moment two agents touch the same module without coordination, you get worse output than a single agent would produce.

This is the production reality check before you go multi-agent. For Spark, decompose by service boundary (each agent owns a service). For full-stack features, the backend/frontend split only works if the API contract is defined before agents start.

Code Review
Anthropic Code Review dispatches parallel agents with under 1% false positive rate

Launched March 9, 2026, Anthropic's Code Review is a managed PR reviewer that runs multiple Claude agents in parallel. Each agent analyzes different sections, they cross-verify findings to filter false positives, then deliver one summary comment plus inline flags on exact lines. Internal testing: 54% of PRs get substantive comments (up from 16%), under 1% of findings marked incorrect by engineers, 84% of 1000+ line PRs get findings averaging 7.5 issues each.

It catches contextual bugs—things syntactically correct but logically wrong. A one-line production change that would have silently broken authentication. A type mismatch in ZFS encryption "silently wiping the encryption key cache on every sync." Average review time ~20 minutes. Cost: $15–25/PR on token usage. Requires Team or Enterprise plan. Will not approve PRs—humans still approve.

You asked about running this locally with a Claude subscription—you can't. It's Team/Enterprise only, cloud-hosted. For your solo setup, the open-source toriihq/claude-review-action (31 stars, MIT) is the closest free alternative. Or build your own with a subagent-based review skill.

Quality Gates
Stop hooks are the missing "done" gate that prevents premature completion

Claude Code's Stop hook fires when Claude finishes responding. Exit code 2 forces Claude to keep working. This is the mechanism for enforcing task completion: a prompt-type hook evaluates whether all tasks are actually done, tests pass, and acceptance criteria are met. If not, it blocks the stop and tells Claude what's left. Agent-type hooks can even grep the codebase to verify claims before allowing completion.

The pattern: "type": "prompt", "prompt": "Evaluate if Claude should stop: $ARGUMENTS. Check if all tasks complete." The LLM responds with {"ok": true} or {"ok": false, "reason": "..."}. Critical: always check a stop_hook_active flag to prevent infinite loops. This transforms "I think I'm done" into "I've verified I'm done."

This directly addresses your regression/waste spiral. Your e2e test hooks catch code-level regressions, but a Stop hook catches the meta-failure: Claude declaring victory with work still undone. Add this as the outermost gate.

Quality Gates
The three-gate pattern (tests + AI review + human approval) is the emerging production standard

Across multiple production write-ups, a consistent quality pattern has emerged for agentic code: Gate 1 is automated tests (pre-commit hooks running integration/e2e suites), Gate 2 is AI code review (either Anthropic's managed Code Review or a subagent-based reviewer), and Gate 3 is human approval (the PR review that no AI should auto-approve). SpecWeave calls these "quality gates" and enforces them before any increment can close. The NousResearch hermes-agent project calls it "fail-closed review."

The critical insight: Gate 1 alone (your current setup) catches syntactic and behavioral regressions but misses logical errors, security issues, and architectural drift. Gate 2 catches what tests don't cover—the "syntactically correct but logically wrong" class. Gate 3 catches what neither automated system can: whether the code matches the actual intent and fits the broader product direction.

You have Gate 1 (e2e hooks). Adding Gate 2 (a code review skill or subagent) is the highest-leverage next step. You can build a simple /review skill that spawns a fresh subagent with "review this diff for logic errors, security, and architectural consistency" and blocks merge until it passes.

Workflow Design
Plan files in the repo beat tickets for agentic development

George Violaris's "agentic development loop" puts a plan file per feature in the repo—not a Jira ticket, not a conversation thread. Each plan file is a contract: objective (observable behavior, not implementation), constraints (what it must not do), dependencies, acceptance criteria, and open questions. Plan files are versioned, diffable, and survive session resets. When the plan changes, you see what changed and when.

His key insight: "The value of agentic coding isn't speed. It's that it forces you to be explicit about what you'd normally leave implicit." The brainstorm phase produces a feature map against your actual codebase context. The plan phase converts it into files detailed enough that an agent unfamiliar with the project could implement without ambiguity. Implementation is the easy part—the plan is where the real work happens.

This directly addresses your pain point about no great process for upfront design. Try it on your next Spark feature: Wispr Flow a 20-min voice dump, have Claude convert it to a plan file with the five sections above, iterate the plan until it's airtight, then implement. The plan file replaces your current gap between "voice spec" and "start coding."

Workflow Design
Spec-driven development is the matured form of vibe coding

Multiple independent sources (Matt Pocock, SpecWeave, Violaris, the Codegen blog) have converged on the same pipeline: brainstorm → spec → plan → implement → review. Matt Pocock's "7 phases" video (32.8K views) structures it as idea → research → prototype → PRD → implementation planning → implementation → review. The Codegen team's "agentic workflows that ship" article identifies context—not the model—as the bottleneck: "The culprit is almost always context: the agent doesn't know enough about the codebase to make good decisions."

The consensus: implementation is the cheapest phase. Spending 60–70% of time on spec and planning produces dramatically better outcomes than jumping to code. The spec is the source of truth; code is its derivative. Bad plans create orders of magnitude more rework than bad code.

You're already doing this intuitively with Wispr Flow voice dumps for feature specs. The next step is formalizing the phases: separate brainstorm (explore the space) from spec (define the contract) from plan (decompose into implementable tasks). Your current skills already support this; the insight is that the pipeline should be explicit and sequential, not collapsed.

Context Management
Hub-and-spoke memory pattern persists architectural decisions across 200+ sessions

A developer building a SaaS product over three months with Claude Code developed the "hub-and-spoke" memory pattern: a central memory index (hub) links to topic-specific memory files (spokes) covering architecture decisions, API contracts, deployment config, and user preferences. Each spoke is a standalone document that can be loaded selectively. The hub file stays under the auto-loaded size limit; spokes are pulled in on demand when relevant to the current task.

The pattern solves the "session 47 doesn't know what session 46 decided" problem without loading everything into context. Key design choices: spokes are organized by topic not chronology, each spoke has a "last verified" date, and stale spokes get pruned during periodic maintenance sessions. The developer reports zero context loss across 200+ sessions on a complex project.

Your setup already uses memory files. The hub-and-spoke formalization is the refinement: keep the auto-loaded MEMORY.md small (just pointers), organize spokes by architectural domain, and add "last verified" dates so you know when a memory was last confirmed true. This is exactly your current system—validate that you're pruning stale entries.

Context Management
StatusLine-based proactive backups catch context before auto-compaction destroys it

The ClaudeFast Code Kit implements a three-file backup system triggered by StatusLine monitoring (the only mechanism that receives live token metrics). The system uses dual triggers: token-based (primary, fires at 50K tokens used then every 10K after) and percentage-based (safety net at 30%, 15%, 5% remaining). Critical detail: remaining_percentage includes a hidden 33K-token autocompact buffer, so you need to subtract it to get actual free-until-compact.

Architecture: backup-core.mjs (shared backup logic), statusline-monitor.mjs (threshold detection + display), and conv-backup.mjs (PreCompact trigger). Backup files are numbered with timestamps. The statusline displays the latest backup path so after compaction you know exactly which file to load. Recommended workflow: when compaction occurs, run /clear to start fresh, then load the backup file.

If you're hitting context limits on large features, this is the insurance policy. The token-based trigger is especially important for 1M context windows where percentage thresholds fire too late. Add this to your hooks and you'll never lose mid-feature context to surprise compaction again.

Hooks & Configuration
HTTP hooks open Claude Code to remote validation services and team-wide policies

As of February 2026, Claude Code hooks support HTTP type handlers that POST event JSON to any endpoint. This enables remote validation services, centralized audit logging, Slack/PagerDuty integrations, and microservice architectures where hook logic runs alongside your API. Key behavior difference from command hooks: non-2xx responses produce non-blocking errors (execution continues). To actually block a tool call, return 2xx with "permissionDecision": "deny" in the body.

Header values support environment variable interpolation, but only for variables in the explicit allowedEnvVars list—unlisted variables resolve to empty strings silently. Hooks are deduplicated by URL (command hooks dedup by command string). Config-only: the /hooks interactive menu only supports command hooks; HTTP hooks must be added by editing settings JSON directly.

For your solo setup, HTTP hooks are overkill. But if you build team tooling for a VP Engineering role, this is how you enforce org-wide policies: one validation endpoint that all Claude Code instances POST to, checking for security patterns, style compliance, or deployment gates centrally.

Model router hook auto-switches tier by task complexity with zero API calls

The tzachbon/claude-model-router-hook (25 stars, MIT) classifies every prompt by complexity using keyword and pattern matching—no API calls, no latency. It runs as a UserPromptSubmit hook: git ops and renames route to Haiku, feature work and debugging to Sonnet, architecture and deep analysis to Opus. A SessionStart hook injects sub-agent model rules so spawned agents also use the right tier. Default behavior is warn-only; set "action": "autoswitch" for automatic model switching.

Classification uses three signal layers: opus keywords (architect, tradeoff, deep dive, complex refactor), haiku patterns (regex matching git commands, renames, formatting), and sonnet patterns (build, implement, fix, debug). Word count thresholds add a coarse signal: >200 words or >100 words with a question mark routes to Opus. Prefix any prompt with ~ to bypass classification entirely.

Direct cost optimization for your setup. Install via claude plugin marketplace add tzachbon/claude-model-router-hook. The sub-agent tier injection is especially valuable—it prevents your subagents from defaulting to Opus for simple file searches. The ~ override gives you an escape hatch when you disagree with classification.

Hooks & Configuration
Async hooks enable non-blocking side effects without slowing the agent

Adding "async": true to any hook runs it in the background without blocking Claude's execution. Released January 2026. This unlocks logging, analytics, backup creation, notifications, and any side-effect that shouldn't add latency to the agent loop. The hook runs concurrently with Claude's next action; results are fire-and-forget.

Not suitable for security blocking (PreToolUse exit code 2 needs to be synchronous), auto-approve decisions, or any hook where Claude needs the result before proceeding. Best combined with the StatusLine backup system: the PreCompact backup hook runs async because the backup doesn't need to block compaction—it just needs to capture state before it's gone.

If you've noticed hooks adding latency to your workflow (especially PostToolUse formatters or loggers), check whether they can be async. Your e2e test hooks must stay synchronous, but logging, backup, and notification hooks should all be async.
