WORKFLOW DESIGN
Spec-driven development is the matured form of vibe coding

Something converged in early 2026 that is worth naming explicitly: multiple independent practitioners, tool builders, and teams arrived at the same development pipeline without coordinating. Matt Pocock published his "7 phases of AI-assisted development" video (32.8K views and climbing). The Codegen team published "Agentic workflows that actually ship." SpecWeave launched with spec-first as its core premise. George Violaris published his plan-file workflow. These are not variations on a theme—they are the same pipeline, discovered independently, which is the strongest possible evidence that the pipeline is correct.

The pipeline, stripped to its essence: brainstorm → spec → plan → implement → review. Five phases, in order, each producing an artifact that feeds the next. The critical insight is that implementation is Phase 4 of 5. Not Phase 1. Not Phase 2. Phase 4. Most of the work—and virtually all of the value—happens before a single line of code is written.

Matt Pocock's 7 phases are the most granular articulation. Phase 1: Idea (what do you want to exist?). Phase 2: Research (what already exists that is relevant? what are the constraints?). Phase 3: Prototype (quick and dirty proof of concept to validate feasibility). Phase 4: PRD—Product Requirements Document (the formal spec: what it does, what it does not do, edge cases, acceptance criteria). Phase 5: Implementation Planning (how to build it: file changes, API contracts, data migrations, task decomposition). Phase 6: Implementation (the actual coding). Phase 7: Review (does it match the PRD?).

Pocock's framing is interesting because he distinguishes between prototype (Phase 3) and implementation (Phase 6). The prototype is disposable. It exists to answer the question "is this feasible?" and nothing else. You throw it away. The implementation is durable. It exists to ship. Conflating the two—which is what "vibe coding" does by default—produces prototypes that accidentally become production code, with all the quality and maintainability problems that implies.

The prototype phase is also where you discover that your idea is impossible, impractical, or needs fundamental rethinking. Better to discover this in 30 minutes with a throwaway prototype than in 3 hours with production-quality code that has to be discarded. The prototype phase is cheap insurance against building the wrong thing well.

The Codegen insight is the most operationally useful finding in the spec-driven literature. Their blog post, based on months of running AI agents against real codebases at scale, identifies the bottleneck: "The culprit is almost always context: the agent doesn't know enough about the codebase to make good decisions." Not model capability. Not prompt quality. Context.

This is a subtle but profound reframing. The conventional wisdom is that better models produce better code. The Codegen finding is that the same model produces dramatically better code when given more relevant context. A mediocre model with perfect context outperforms a superior model with incomplete context. This means the highest-leverage investment is not in model selection or prompt engineering—it is in context engineering. How do you get the right information in front of the agent at the right time?

The spec is the primary context delivery mechanism. A well-written spec tells the agent: what the feature should do (behavior), what it should not do (constraints), what already exists that is relevant (dependencies and interfaces), what "done" means (acceptance criteria), and what is unknown (open questions that need human input). Without the spec, the agent has to infer all of this from the codebase alone, which is like asking a new hire to build a feature by reading the source code with no onboarding, no requirements document, and no one to ask questions.

The Codegen team's practical recommendation: before starting any implementation, run an explicit "context gathering" phase where the agent reads relevant files, identifies interfaces, maps dependencies, and produces a summary of what it has learned. This summary becomes the foundation of the spec. The agent is better at reading code than writing requirements, so use it to read the codebase and then have the human validate and extend the summary into a full spec.

Why implementation is the cheapest phase. This claim sounds provocative until you quantify the costs. A Phase 6 failure (implementation bug) costs one fix cycle: the agent or human identifies the bug, fixes it, and re-runs tests. Elapsed time: 5–30 minutes. A Phase 5 failure (bad plan) costs a rewrite of the implementation: the plan decomposed the work incorrectly, and the agent built components that do not fit together. Elapsed time: 1–3 hours. A Phase 4 failure (bad spec) costs a rewrite of the plan and the implementation: the spec defined the wrong behavior, and everything built on top of it is wrong. Elapsed time: 3–8 hours. A Phase 1 failure (bad idea) costs everything: you built the wrong thing entirely. Elapsed time: days.

The cost multiplier is roughly 10x per phase. A bug in implementation costs 1x. A bug in the plan costs 10x. A bug in the spec costs 100x. A bug in the idea costs 1000x. This is not new—it is the same cost curve that waterfall advocates have cited for decades. What is new is that agentic coding makes the cost curve steeper, because agents implement faster than humans. An agent that implements a bad spec produces a larger volume of wrong code in less time, which means more code to review, more code to discard, and more time wasted. Speed without direction is expensive.

Separating brainstorm from spec from plan. The three phases before implementation serve different cognitive purposes and should not be collapsed.

Brainstorm (Phase 1–2 in Pocock's model) is divergent. You are exploring the space. What could this feature be? What are the options? What are the trade-offs? What has been tried before? What are the constraints from the existing codebase? The output is a feature map: a messy, incomplete, exploratory document that captures the possibility space. The brainstorm is intentionally unconstrained. You are not making decisions; you are generating options.

Spec (Phase 3–4) is convergent. You are collapsing the possibility space into a single, specific definition. The feature does X. It does not do Y. The edge cases are handled as follows. The acceptance criteria are these specific, testable conditions. The spec is a contract. It should be precise enough that two independent agents, implementing the spec without communicating, would produce functionally equivalent code. If the spec is ambiguous enough that two implementations could diverge meaningfully, the spec is not done.

Plan (Phase 5) is decomposition. The spec says what to build. The plan says how to decompose the work into implementable units. Which files to change. What order to make the changes. Where the integration points are. What can be parallelized and what must be sequential. The plan is specific to the codebase—it references real file paths, real function names, real data models. The spec is abstract (behavior); the plan is concrete (changes).

Collapsing brainstorm and spec is the most common mistake. The developer starts writing a spec before fully exploring the space, commits to the first approach that seems reasonable, and misses better alternatives or critical constraints. The result is a spec that is precise but wrong—it precisely defines a suboptimal or infeasible approach. Separating brainstorm from spec gives you permission to explore without committing, which produces better specs.

Collapsing spec and plan is the second most common mistake. The developer writes a spec that is half behavior definition and half implementation plan. The behavior and the implementation become entangled, and changing one requires changing the other. A clean spec describes what the feature does without specifying how. A clean plan describes how to implement the spec without redefining what. This separation lets you change the plan (different implementation approach) without changing the spec (same behavior), and vice versa.

How to structure each phase with an agent. Brainstorm: give the agent the codebase context and a rough description of the feature. Ask it to produce a feature map: what exists, what changes, what is created, what are the options, what are the risks. Review the feature map. Add your own knowledge. Iterate until the space feels fully explored.

Spec: take the feature map and instruct the agent to produce a spec with five sections: objective (observable behavior), constraints (what it must not do), dependencies (what it relies on), acceptance criteria (testable conditions for done), and open questions (unresolved decisions). Review the spec against the feature map. Every option from the brainstorm should either be reflected in the spec or explicitly rejected. Every constraint from the codebase should appear in the constraints section. Iterate until the spec is tight enough that implementation is mechanical.

Plan: take the spec and instruct the agent to produce an implementation plan. Which files change. What functions are added, modified, or removed. What the data flow looks like. What the test strategy is. What order the changes should be made. The plan should be specific enough that a different agent (or a junior developer) could follow it step by step. Review the plan against the spec. Every acceptance criterion should map to at least one change in the plan. Every change in the plan should trace back to something in the spec.

Implementation: give the agent the spec and the plan. Instruct it to implement the plan, referencing the spec for behavior and the plan for structure. The agent should not need to make design decisions—those were made in earlier phases. If it encounters an ambiguity, it should flag it rather than guess.

Review: compare the implementation against the spec and plan. Does the code satisfy every acceptance criterion? Does it violate any constraint? Are there changes that do not trace back to the plan? Are there plan items that were not implemented?

The spec-driven pipeline and vibe coding are not opposites. Vibe coding is Phase 1: you have an idea, you start building, you see what happens. Spec-driven development acknowledges that Phase 1 is valuable for exploration but recognizes that exploration is not construction. The pipeline takes the energy and intuition of vibe coding (brainstorm, prototype) and channels it through the discipline of spec writing (spec, plan) before committing to implementation. Spec-driven development is vibe coding that has learned from its mistakes.

The community is converging on this pipeline because it is the natural consequence of a simple observation: agents are excellent implementers and terrible requirements gatherers. They will build exactly what you tell them to build, with impressive speed and reasonable quality. The problem is never "the agent can't build it." The problem is always "the agent built the wrong thing" or "the agent built it the wrong way." The spec-driven pipeline addresses both by front-loading the decisions that determine what "the right thing" and "the right way" mean, and encoding those decisions in artifacts that the agent can reference during implementation.

The maturity progression is clear: vibe coding (just build) to plan files (build against a contract) to spec-driven development (brainstorm, spec, plan, build, review as explicit sequential phases). Each step adds discipline to the phase where discipline has the highest return. If you are already using plan files, the next step is formalizing the brainstorm and spec phases as separate, reviewable artifacts.
