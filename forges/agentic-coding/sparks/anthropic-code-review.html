CODE REVIEW
Anthropic Code Review dispatches parallel agents with under 1% false positive rate

Anthropic Code Review shipped on March 9, 2026, and the architecture is worth dissecting because it represents the first production-grade implementation of multi-agent code analysis where the agents coordinate to suppress false positives rather than amplify them. The system runs as a GitHub integration: when a PR is opened or updated, it dispatches multiple Claude agents in parallel, each assigned to analyze different sections of the changeset. These agents produce independent findings, which are then cross-verified against each other and against the broader codebase context. Findings that survive cross-verification are severity-ranked, and the final output is one summary comment on the PR plus inline comments on specific lines. The entire pipeline runs in roughly 20 minutes for a typical PR.

The numbers from internal testing are the headline story. 54% of PRs receive substantive comments—comments that identify real issues requiring action, not style nitpicks or formatting suggestions. This is up from 16% in earlier iterations, which tells you the cross-verification filter is doing serious work. Under 1% of findings are marked incorrect by the reviewing engineer, which is a false positive rate low enough to build trust. For large PRs (1000+ lines), 84% receive findings with an average of 7.5 issues per PR. For small PRs (under 100 lines), only 31% receive findings, which is the correct behavior—small, focused changes simply have fewer opportunities for error.

The class of bug it catches is the one that matters most: contextual errors. Code that is syntactically valid, passes type checking, and would even pass most unit tests, but is logically wrong given the broader system context. The canonical example from internal testing is a one-line production change that would have silently broken authentication. The change was a refactor of a function signature—perfectly valid in isolation. But the function was called in an auth middleware path where the argument order mattered for security, and the refactored signature swapped two parameters of the same type. No compiler would catch it. No type checker would catch it. A unit test for the function in isolation would pass. But the auth check would silently succeed for any input.

A second example that circulated in the engineering community: a type mismatch in a ZFS encryption module that would have silently wiped the encryption key cache on every sync operation. The code compiled, the tests passed (because the test fixture used a fresh cache that was empty anyway), but in production with a populated cache, every sync would destroy the keys. The Code Review agent caught this because it analyzed the interaction between the sync function and the cache lifecycle, not just the diff in isolation.

The architecture deserves closer examination. The parallel dispatch is not random—agents are assigned sections based on a dependency graph extracted from the diff. Files that import from each other get analyzed by the same agent or by agents that share their findings before cross-verification. This means the system understands that a change to auth/middleware.ts is related to a change to auth/types.ts even if they appear in different hunks. The cross-verification step is where most false positives die: Agent A flags something, Agent B has context that explains why it is actually correct, the finding gets dropped. This is fundamentally different from a single-pass review where the reviewer has to hold all context simultaneously.

The severity ranking uses four tiers: critical (will cause data loss, security breach, or crash in production), high (logic error that will produce incorrect results), medium (code smell, performance issue, or maintainability concern), and low (style, naming, documentation). The summary comment includes a breakdown by severity. Inline comments are placed on the exact line where the issue is, with an explanation that references the broader context—not just "this looks wrong" but "this conflicts with the invariant established in src/core/cache.ts:142."

Cost and access constraints matter. At $15–25 per PR in token costs, this is not cheap for high-volume repositories. A team pushing 20 PRs per day is looking at $300–500/day, or roughly $7,500–12,500/month. That is a line item that needs explicit budgeting. The system requires a Team or Enterprise plan—it does not work with individual Pro subscriptions, and it cannot be run locally. This is a deliberate architectural choice: the multi-agent coordination runs on Anthropic's infrastructure, not yours.

Critically, it will not approve PRs. It comments, it flags, it severity-ranks, but the green checkmark still requires a human. This is an opinionated design decision that some teams will find frustrating (why can it not auto-approve if it finds zero issues?) but it is the correct one. The false negative rate—things it misses—is not zero, and auto-approval would create a false sense of security. The system is a reviewer, not an approver.

For developers who cannot access Code Review (solo developers, open-source projects, teams on lower-tier plans), the open-source alternatives are worth evaluating. toriihq/claude-review-action (31 stars, MIT license) is the most direct equivalent: a GitHub Action that runs Claude against your PR diff and posts comments. It is single-agent, not multi-agent, so it lacks the cross-verification that drives the low false positive rate. But for straightforward reviews it is functional and free. altinukshini/claude-code-pr-reviewer takes a similar approach with slightly different prompt engineering.

The Claude Code GitHub Action (officially maintained by Anthropic, free for Claude Code subscribers) offers a lighter-weight review mode. It runs a single Claude Code session against the PR diff with a review-oriented system prompt. The analysis is less thorough than dedicated Code Review—no parallel agents, no cross-verification—but it catches surface-level issues and is a reasonable Gate 2 for teams that cannot justify the $15–25/PR cost.

Building your own multi-agent reviewer is feasible if you understand the subagent pattern. The approach: write a Claude Code skill that receives a diff, splits it into sections based on file dependencies, spawns one subagent per section with instructions to review that section for logic errors and security issues, collects findings, runs a final subagent that cross-verifies all findings against each other, and outputs a consolidated report. The key implementation detail is the cross-verification step—without it, you get the 16% substantive rate, not the 54% rate. The cross-verifier needs access to all findings plus the full codebase context to distinguish genuine issues from false positives.

A minimal implementation looks like this:

// Skill: /review-pr
// 1. Get the diff
const diff = await getDiff();
// 2. Split by dependency clusters
const sections = clusterByImports(diff);
// 3. Spawn parallel review agents
const findings = await Promise.all(
  sections.map(s => spawnSubagent({
    prompt: `Review this diff section for logic errors,
    security issues, and architectural inconsistencies.
    Codebase context: ${s.relatedFiles}`,
    diff: s.hunks
  }))
);
// 4. Cross-verify
const verified = await spawnSubagent({
  prompt: `Given these findings from parallel reviewers,
  eliminate false positives by cross-referencing.
  Only keep findings that are genuinely incorrect
  given the full context.`,
  findings: findings.flat(),
  fullDiff: diff
});
// 5. Format and post
postReviewComments(verified);

The gap between this DIY approach and Anthropic's managed service is primarily in the dependency graph construction (theirs is more sophisticated) and the training data for cross-verification (theirs has been tuned on millions of internal reviews). But for a team that needs code review AI and cannot justify the managed service cost, a subagent-based skill that follows this pattern will catch the majority of contextual bugs that a single-pass review misses.

One final operational note: Code Review's 20-minute execution time means it does not work well as a blocking gate for fast-iteration workflows. If your team expects review feedback within 2–3 minutes of pushing, this will feel slow. The pattern that works is: push PR, continue working on something else, come back to the review comments when they arrive. Treat it like a human reviewer who takes 20 minutes to respond, not like a linter that responds in seconds.

If you are on Team/Enterprise, enable Code Review and let the numbers speak for themselves. If you are solo or on Pro, build the subagent review skill above—it will not match the managed service's false positive rate but it will catch the contextual bugs that tests miss, which is the entire value proposition.
