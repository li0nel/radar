Orchestration
ACP protocol and ACPX replace PTY scraping for cross-vendor agent orchestration

The fundamental problem with running multiple coding agents today is that they cannot talk to each other. You launch Claude Code in one terminal, Codex in another, Gemini CLI in a third, and they are islands. The only way to pass information between them is through you—copy-pasting output from one into the prompt of another, or through shared files on disk with no coordination protocol. Every team that has tried multi-agent development at scale has independently discovered that this is the bottleneck, not model quality or context windows or cost.

PTY scraping was the first attempt to solve this. Tools like expect, script, and custom Node wrappers would attach to the agent's terminal, parse the ANSI output stream, extract structured information from what is fundamentally unstructured text, and pipe it somewhere else. This works until it doesn't. Terminal output is a presentation format, not a data interchange format. Agents reformat their output, add color codes, insert progress spinners, wrap lines differently based on terminal width, and occasionally emit binary content that breaks parsers. Every agent update risks breaking the scraper. The approach is inherently fragile because it violates the abstraction boundary—you're reading the rendering layer instead of the semantic layer.

The Agent Communication Protocol (ACP) from Zed Industries is the first serious attempt to define a structured communication layer between coding agents. Important disambiguation: this is not BeeAI's Agent Communication Protocol, which was an earlier, now-deprecated effort focused on general-purpose agent interoperability. Zed's ACP is specifically designed for coding agents and is built on JSON-RPC 2.0 over stdio—the same transport mechanism that LSP uses. This is not a coincidence. The ACP designers explicitly modeled the protocol after LSP because it solved the same M×N integration problem: before LSP, every editor needed a custom integration for every language. Before ACP, every orchestrator needs a custom integration for every agent.

The protocol defines a small set of methods: acp/initialize for capability negotiation, acp/execute for sending tasks, acp/status for polling progress, acp/result for retrieving output, and acp/cancel for aborting. Each agent implements an ACP server (a thin wrapper around its existing CLI), and orchestrators connect to it as ACP clients. The capability negotiation step is critical—it lets the orchestrator discover what each agent can do (file editing, web search, code execution, etc.) and route tasks accordingly.

ACPX is the reference CLI implementation of an ACP orchestrator. Published by openclaw under MIT license on GitHub, it currently supports 15+ agents including Claude Code, Codex, Gemini CLI, Aider, Continue, Cursor Agent, Kiro, Windsurf, and several others. Installation is straightforward:

npm install -g @openclaw/acpx

Or if you prefer not to install globally:

npx @openclaw/acpx init

The init command creates a .acpxrc.json in your project root that declares which agents are available and how to launch them. A minimal configuration for Claude Code plus Codex looks like this:

{
  "agents": {
    "claude": {
      "command": "claude",
      "args": ["--acp"],
      "capabilities": ["edit", "search", "execute"],
      "model": "claude-sonnet-4-20250514"
    },
    "codex": {
      "command": "codex",
      "args": ["--acp"],
      "capabilities": ["edit", "search"],
      "model": "o3"
    }
  },
  "sessions": {
    "persist": true,
    "crashReconnect": true,
    "timeout": 300
  }
}

The --acp flag tells each agent to start in ACP server mode instead of interactive mode. In this mode, the agent listens on stdin for JSON-RPC messages and responds on stdout. No terminal UI, no ANSI codes, no interactive prompts—pure structured data.

Named sessions are one of ACPX's most practical features. When you start an orchestration session, ACPX assigns it a name (auto-generated or specified with --session) and persists the session state to .acpx/sessions/. If an agent crashes—which happens more often than vendors admit, especially under memory pressure from large codebases—ACPX detects the disconnection, relaunches the agent, re-initializes the ACP handshake, and replays enough context for the agent to resume. The "crashReconnect": true setting enables this. Without it, a single agent crash in a multi-agent workflow means manually restarting everything and figuring out where you left off.

Fire-and-forget mode (acpx fire <agent> <task>) is for tasks where you don't need to monitor progress interactively. You dispatch a task to an agent and ACPX logs the result to a session file. This is useful for parallelizing independent workstreams: fire off a test-writing task to one agent, a documentation task to another, and a refactoring task to a third. Check results when they complete. The alternative—sequentially waiting for each agent to finish before starting the next—wastes the time savings that multi-agent orchestration is supposed to provide.

The practical workflow with Claude Code as the primary agent and Codex or Gemini as secondary agents looks like this: you work interactively with Claude Code on the main feature branch. When you encounter an independent sub-task (writing tests for a module, updating documentation, refactoring a utility file that isn't touched by your current work), you dispatch it to a secondary agent via ACPX. Claude Code continues working on the main task while the secondary agent works in parallel. When the secondary agent completes, ACPX notifies you (or, if you've configured it, the notification arrives in cmux as a colored attention ring).

# Start interactive session with Claude as primary
acpx session start --name feature-auth --primary claude

# Dispatch parallel tasks to secondary agents
acpx fire codex "Write unit tests for src/auth/oauth.ts \
  covering all error paths. Output to tests/auth/oauth.test.ts"
acpx fire gemini "Update API documentation in docs/auth.md \
  to reflect the new OAuth flow in src/auth/oauth.ts"

# Check status of all agents
acpx status

# Get results when done
acpx result codex --session feature-auth

Now, the political situation. Anthropic's position on ACP is complicated. GitHub issue #6686 on the Claude Code repository requested native ACP support. It accumulated 439 thumbs-up reactions, making it one of the most-requested features. Anthropic closed it. The official response cited concerns about protocol stability (ACP is still at version 0.3), security implications of inter-agent communication (an agent could be tricked into executing tasks from an untrusted source), and the preference for first-party orchestration via Agent Teams rather than third-party protocol adoption.

The community's workaround is zed-industries/claude-agent-acp, an adapter that wraps Claude Code's CLI in an ACP-compliant server. It intercepts Claude Code's stdout, parses the structured JSON that Claude Code already emits in --output-format json mode, and translates it into ACP responses. It's not as clean as native support would be—there's a translation layer that adds latency and occasionally loses metadata—but it works well enough for the common cases. Installation:

npm install -g @zed-industries/claude-agent-acp

# Then in .acpxrc.json, replace the claude agent entry:
{
  "agents": {
    "claude": {
      "command": "claude-agent-acp",
      "args": ["--model", "claude-sonnet-4-20250514"],
      "capabilities": ["edit", "search", "execute"]
    }
  }
}

The M×N problem that ACP solves is worth understanding precisely because it's the same problem LSP solved. Before LSP, if you had M editors and N languages, you needed M×N integrations (Vim+Python, Vim+Go, Emacs+Python, Emacs+Go, etc.). LSP reduced this to M+N: each editor implements one LSP client, each language implements one LSP server. ACP does the same for agent orchestration: if you have M orchestrators and N agents, ACP reduces the integration surface from M×N to M+N. Each orchestrator implements one ACP client, each agent implements one ACP server.

This matters more than it appears. Right now, the agent landscape is fragmenting. New agents ship monthly. Without a protocol, each new agent requires custom integration work in every orchestrator. With ACP, a new agent just needs to implement the ACP server interface and it's immediately usable by every ACP-compatible orchestrator. The network effect is the same one that made LSP successful—it took about three years from LSP's introduction for it to become the universal standard. ACP is at month eight.

Limitations to be aware of: ACP currently has no authentication mechanism, which means any process that can write to the agent's stdin can send it commands. In a multi-user environment, this is a security concern. The protocol has no built-in rate limiting, so a runaway orchestrator can flood an agent with requests. There's no standardized error taxonomy—agents can return arbitrary error codes, and orchestrators have to handle them case-by-case. And the capability negotiation, while useful, is self-reported by agents—there's no verification that an agent actually supports the capabilities it claims.

The spec is also silent on resource contention. If two orchestrators connect to the same agent instance, ACP doesn't define who wins. In practice, agents are single-tenant (one orchestrator per agent instance), and you launch separate agent instances for separate orchestration sessions. ACPX handles this automatically by managing the lifecycle of agent processes, but if you're building custom tooling on top of ACP, you need to manage this yourself.

Despite these limitations, ACP and ACPX represent the first viable path from single-agent workflows to cross-vendor multi-agent orchestration. The alternative is vendor lock-in to a single agent's proprietary multi-agent system (Anthropic's Agent Teams, which only orchestrates Claude Code instances) or continued reliance on PTY scraping and clipboard-based coordination. For teams that want to use the best agent for each task—Claude for architecture, Codex for test generation, Gemini for documentation—ACP is currently the only game in town.

If you're running more than one coding agent regularly, ACPX is the first tool to evaluate. The named sessions and crash reconnect alone justify the setup time. Start with two agents (Claude + one other) and one fire-and-forget task per session to build intuition before attempting full multi-agent orchestration.
