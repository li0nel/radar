Hooks & Configuration
Model router hook auto-switches tier by task complexity with zero API calls

Claude Code's default behavior is to use whatever model you've configured for everything. Ask it to rename a variable, it uses Opus. Ask it to architect a distributed system, it uses Opus. Ask it to run git status, it uses Opus. This is wasteful in the precise, measurable sense: Haiku is 50x cheaper than Opus per token and responds 5x faster. For a git status invocation, you are paying Opus prices and waiting Opus latency for a task that Haiku handles identically. The model router hook by tzachbon (25 stars, MIT license) fixes this with two shell scripts and zero additional API calls.

The system consists of two hook scripts that operate at different lifecycle points. The first, session-init.sh, fires on SessionStart and injects model-tier routing rules into the session context. The second, model-router-hook.sh, fires on UserPromptSubmit and classifies each prompt against a three-tier model hierarchy. The classification is entirely local—regex matching and word counting in bash. No LLM call, no API request, no network dependency. The decision happens in single-digit milliseconds.

The three-tier hierarchy maps directly to Anthropic's model lineup: Haiku for mechanical tasks, Sonnet for standard development work, Opus for deep architectural reasoning. The insight is that most agentic coding sessions contain a mix of all three complexity levels, and the distribution is heavily skewed toward the low end. In a typical feature implementation session, roughly 60% of prompts are Haiku-tier (run this command, rename this, format that), 30% are Sonnet-tier (implement this function, fix this bug, write these tests), and 10% are Opus-tier (analyze this architecture, evaluate these tradeoffs, design this system). The model router captures that distribution and routes accordingly.

The classification engine uses three signal layers, evaluated in priority order. First, opus keywords: if the prompt contains words like "architect," "tradeoff," "trade-off," "deep dive," "system design," "evaluate alternatives," or "analyze the codebase," it routes to Opus. These are terms that signal open-ended reasoning tasks where model quality directly impacts output quality. Second, haiku patterns: regex-based matching for git commands (git (status|log|diff|add|commit|push|pull|fetch|branch|checkout|merge|rebase|stash|tag)), file renames, formatting requests, and other mechanical operations. Third, sonnet patterns: keywords like "build," "implement," "fix," "debug," "write," "create," "add feature," and "refactor." Anything that doesn't match an explicit pattern defaults to Sonnet—a sensible middle ground.

Word count acts as a secondary signal that overrides keyword matching in specific cases. Prompts longer than 200 words are routed to Opus regardless of keyword content, on the theory that lengthy prompts describe complex tasks even if they don't use architecture-specific terminology. A developer writing three paragraphs about a concurrency problem might never say "architect" but is clearly describing an Opus-tier task. Conversely, prompts shorter than 60 words that match a haiku pattern are confirmed as Haiku-tier. The thresholds are configurable in .claude/model-router.json:

{
  "thresholds": {
    "opusWordCount": 200,
    "haikuMaxWords": 60
  },
  "tiers": {
    "opus": {
      "keywords": ["architect", "tradeoff", "deep dive", "system design"],
      "minWords": 200
    },
    "haiku": {
      "patterns": ["^git\\s+(status|log|diff|add|commit)", "rename", "format"],
      "maxWords": 60
    },
    "default": "sonnet"
  }
}

The default behavior is warn-only mode. The hook classifies the prompt and outputs a message like [model-router] Recommended tier: haiku (matched: git command pattern), but does not actually switch the model. This is the safe starting point—you observe the classifications for a few days, check whether they match your intuition, and tune the patterns. When you're satisfied, set "action": "autoswitch" in the configuration to enable automatic model switching. The hook's UserPromptSubmit handler returns a JSON response that sets the model for that specific invocation without changing the session-level default.

The tilde prefix (~) is the escape hatch. Prefixing any prompt with ~ bypasses the router entirely and uses whatever model is configured as the session default. This is for the cases where the router gets it wrong—when you type a short prompt that matches a haiku pattern but actually requires deep reasoning, or when you want Opus on a task the router would classify as Sonnet. The bypass is logged, so you can review bypassed prompts to identify patterns the router should learn.

The session-init.sh script is where the real leverage lives, and it's the feature that distinguishes this from a naive prompt classifier. On SessionStart, the script injects model-tier rules not just for the main session but for all sub-agents. This is critical because Claude Code spawns sub-agents for many operations—file search, code analysis, test execution—and by default, sub-agents inherit the session's model. Without the session-init injection, a sub-agent spawned to do a simple Glob search would use Opus. With it, the sub-agent receives instructions to use Haiku for mechanical tool operations and Sonnet for anything requiring judgment.

The sub-agent injection works by appending model-routing rules to the session's system context via the hook's systemMessage return field. The injected text reads approximately: "For sub-agent tasks: use haiku for file search, glob, grep, and read operations. Use haiku for git commands and simple file operations. Use sonnet for code analysis, implementation, and debugging. Use opus only when explicitly instructed or when the task involves architectural decisions across multiple systems." Claude respects these instructions because they arrive as system-level context, which has higher priority than user-level instructions in Claude's attention hierarchy.

This is where the cost savings compound. In a typical agentic coding session, the main Claude instance might process 20 user prompts. But those 20 prompts can spawn 200+ sub-agent invocations—searching for files, reading code, running tests, checking types. If each sub-agent invocation uses Opus, you're paying Opus rates for 200 operations where Haiku would produce identical results. The session-init injection routes those 200 operations to Haiku, and you pay Opus rates only for the 2-3 prompts that actually require architectural reasoning. The math works out to roughly a 70% cost reduction on a typical session, with no measurable quality degradation on any individual task.

Installation is available through the Claude Code plugin marketplace (search "model-router") or manually by cloning the repository and adding the hook definitions to your settings:

{
  "hooks": {
    "SessionStart": [
      {
        "type": "command",
        "command": "bash ~/.claude/hooks/model-router/session-init.sh"
      }
    ],
    "UserPromptSubmit": [
      {
        "type": "command",
        "command": "bash ~/.claude/hooks/model-router/model-router-hook.sh"
      }
    ]
  }
}

Logging goes to ~/.claude/hooks/model-router-hook.log. Each entry records the timestamp, the prompt (truncated to 100 characters), the matched pattern (if any), the word count, and the routing decision. This log is your primary debugging tool when the router makes a wrong call. Review it weekly for the first month, tune patterns, and adjust thresholds. After the initial tuning period, the router stabilizes and the log becomes a cost-auditing tool—you can calculate exactly how many tokens were routed to each tier and estimate the cost savings.

Limitations are real but bounded. The classifier is lexical, not semantic. A prompt like "can you quickly architect a rename script" contains both the opus keyword "architect" and the haiku keyword "rename." The priority order (opus > haiku > sonnet) means this routes to Opus, which is arguably wrong for what's actually a simple scripting task. The word count heuristic helps in some cases but not all. For teams with specialized vocabularies—a database team where "schema" is a common word in mechanical operations, not just architectural ones—the default keyword lists will need customization.

The router also cannot classify based on codebase context. A prompt like "fix the bug" might be Haiku-tier (typo fix) or Opus-tier (concurrency race condition) depending on what "the bug" is, and the router has no way to know. It classifies based solely on the prompt text, not on the files Claude is looking at, the conversation history, or the nature of recent errors. A more sophisticated version could incorporate tool-use history (if the last 10 tool invocations were all in the same file, the task is probably focused and Sonnet-appropriate; if they spanned 30 files, it's probably architectural and Opus-appropriate), but that's future work.

Despite these limitations, the model router represents a pragmatic approach to one of the real cost problems in agentic coding: the mismatch between task complexity and model capability. You don't need a perfect classifier. You need one that's right 80% of the time and has an escape hatch for the other 20%. At 70% cost reduction for zero quality loss on correctly-routed tasks, the ROI is immediate.

Install in warn-only mode first and run it for a week before enabling autoswitch. Review the log at ~/.claude/hooks/model-router-hook.log to verify classifications match your expectations. The sub-agent injection via session-init.sh delivers most of the cost savings—even if you never enable autoswitch on user prompts, the sub-agent routing alone is worth the setup.
