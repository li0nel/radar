Hooks & Configuration
HTTP hooks open Claude Code to remote validation services and team-wide policies

Command hooks run a local process. HTTP hooks POST a JSON payload to a URL. That single difference—local process vs. network request—changes the entire operational model. With command hooks, every developer's machine must have the hook script installed, kept in sync, and running the same version. With HTTP hooks, the validation logic lives on a server your team controls, and every Claude Code instance in the organization hits the same endpoint. One deployment updates the policy for everyone. Released in February 2026, HTTP hooks are the mechanism that makes Claude Code governable at organizational scale.

The semantics are deliberately different from command hooks, and the differences matter for correctness. A command hook that exits with code 2 blocks execution—hard stop, Claude sees the denial. An HTTP hook that returns a non-2xx status code does not block execution. This is the single most misunderstood aspect of HTTP hooks. A 500, a 503, a timeout, a DNS failure—none of these prevent Claude from proceeding. The design rationale is resilience: if your validation server is down, you don't want every developer on your team locked out of Claude Code. Network failures are treated as non-blocking errors. Claude sees the error message but continues working.

To actually block an action via HTTP hook, the endpoint must return a 2xx status code with a JSON body that contains an explicit deny. The response body must include "decision": "block" (or the equivalent field your hook configuration expects). This is a critical design pattern: blocking is an affirmative action, not a failure mode. Your server must be healthy enough to return a 200 and explicitly say "no." This means your validation service needs to be reliable in both directions—it must be up to approve things and up to deny them.

The response handling matrix is worth committing to memory:

| Response             | Claude Sees           | Execution |
|----------------------|-----------------------|-----------|
| 2xx + empty body     | Nothing               | Continues |
| 2xx + plain text     | Text as info message  | Continues |
| 2xx + JSON (allow)   | Nothing or message    | Continues |
| 2xx + JSON (block)   | Block reason message  | BLOCKED   |
| Non-2xx (any body)   | Error info message    | Continues |
| Timeout              | Timeout warning       | Continues |
| Connection refused   | Connection error      | Continues |

The only path to blocking is the fourth row. Every failure mode defaults to allowing execution. This is a permissive-by-default design, and it's the right one for a development tool—you don't want your CI-style validation infrastructure to become a single point of failure for every developer's coding session.

Configuration lives in .claude/settings.json (project-level) or ~/.claude/settings.json (user-level). An HTTP hook definition looks like this:

{
  "hooks": {
    "PreToolUse": [
      {
        "type": "http",
        "url": "https://hooks.internal.corp.com/claude/validate",
        "headers": {
          "Authorization": "Bearer $CLAUDE_HOOK_TOKEN",
          "X-Team-Id": "$TEAM_ID"
        },
        "allowedEnvVars": ["CLAUDE_HOOK_TOKEN", "TEAM_ID"],
        "timeout": 5000
      }
    ]
  }
}

The allowedEnvVars field is the security mechanism that prevents HTTP hooks from becoming an accidental secret exfiltration vector. When Claude Code interpolates environment variables in the headers object, it only interpolates variables explicitly listed in allowedEnvVars. Any $VAR reference not in the allowlist resolves to an empty string, silently. No error, no warning. This is intentional—it prevents a misconfigured hook from accidentally sending $AWS_SECRET_ACCESS_KEY or $GITHUB_TOKEN to an endpoint. The silence is debatable (a warning would help debugging), but the security tradeoff favors quiet failure.

The practical implication: if your HTTP hook's authorization suddenly stops working, check allowedEnvVars first. The most common failure mode is adding a header that references a new environment variable without adding it to the allowlist. The header gets sent with an empty value, the server rejects it, but because non-2xx responses are non-blocking, Claude proceeds without validation. You've silently disabled your policy enforcement. Monitoring your hook endpoint's request logs is the only reliable way to catch this.

HTTP hooks are deduplicated by URL. If two different hook configurations in the same lifecycle event point to the same URL, Claude Code sends the request once. This is relevant when project-level and user-level settings both define hooks for the same endpoint—common when a team's shared configuration overlaps with a developer's personal configuration. The deduplication is purely URL-based; different headers or different timeout values on the same URL still result in a single request (using the configuration from the higher-priority source).

There is no /hooks menu support for HTTP hooks. Unlike command hooks, which can be created and managed through Claude Code's interactive /hooks slash command, HTTP hooks are config-only. You must edit .claude/settings.json directly. This is a significant ergonomic gap for teams trying to roll out HTTP hooks—developers who are used to the /hooks UI for managing command hooks will look for HTTP hooks there and not find them. The likely reason is that HTTP hooks are aimed at platform teams who manage configuration centrally, not individual developers who configure hooks interactively.

The event JSON payload that gets POSTed to your endpoint follows the same structure as command hook input, but delivered as an HTTP body instead of stdin. For a PreToolUse event on a Bash tool invocation:

{
  "event": "PreToolUse",
  "tool": "Bash",
  "input": {
    "command": "rm -rf /tmp/build",
    "description": "Clean build directory"
  },
  "session": {
    "id": "sess_abc123",
    "cwd": "/home/dev/project"
  }
}

Your endpoint receives this, applies whatever validation logic you want, and returns a decision. A blocking response looks like:

{
  "decision": "block",
  "reason": "Destructive rm -rf commands require explicit approval. Use the cleanup script at ./scripts/clean.sh instead."
}

The reason field is surfaced to Claude as the block message, and Claude will inform the user why the action was prevented. This creates a feedback loop: the developer sees the policy, learns about the preferred alternative, and adjusts their behavior. Over time, the hook fires less often because developers internalize the norms. This is the organizational value of HTTP hooks—they encode institutional knowledge into infrastructure.

The canonical use case is routing Bash commands through a validation endpoint. A team-wide HTTP hook on PreToolUse intercepts every tool invocation, checks if it's a Bash command, and applies a policy: no rm -rf outside of designated directories, no curl to external URLs in production repositories, no docker commands without the team's standard resource limits, no git push --force to protected branches. The endpoint is a small Go or Python service with a rule engine that takes maybe a day to write and runs on a shared server. One service, enforced everywhere.

Centralized audit logging is the second major use case. An HTTP hook on PostToolUse posts every tool invocation and its result to an audit service. This gives you a complete record of what Claude did across every developer session in your organization. For regulated industries—finance, healthcare, government contracting—this is not optional. The audit trail must exist, and it must be tamper-resistant (which means it can't live on the developer's local machine where they could delete hook logs). HTTP hooks posting to a centralized, append-only audit service satisfy this requirement.

Slack and PagerDuty integration is straightforward. An HTTP hook on the Notification event can forward Claude Code notifications to a Slack channel. A hook on Stop can alert a PagerDuty endpoint if Claude stops unexpectedly during an attended-off workflow. These are fire-and-forget side effects—you don't need the response, and you don't want a Slack API timeout to block Claude. Combine with "async": true for non-blocking delivery.

Microservice architectures benefit from HTTP hooks as the glue layer between Claude Code and internal services. A PreToolUse hook can check a deployment service's lock status before allowing file edits in a repository that's mid-deploy. A SessionStart hook can register the developer's session with a coordination service that prevents two developers from having Claude Code work on the same files simultaneously. A PostToolUse hook on file writes can trigger incremental builds on a build farm. The pattern is always the same: Claude Code emits an event, the event hits a URL, the URL triggers business logic.

When HTTP hooks are overkill: solo developers, small teams where everyone is in the same room, projects where the validation logic is simple enough to express as a shell script. A command hook that greps for dangerous patterns in Bash commands is faster to set up, has zero network latency, works offline, and doesn't require maintaining a server. The crossover point is roughly when you have more than five developers using Claude Code, or when your validation logic needs to query external state (deployment status, feature flags, access control lists) that doesn't exist on the developer's local machine.

When HTTP hooks are essential: any team larger than ten developers, any organization with compliance requirements, any environment where policy must be enforced uniformly and cannot depend on developers installing the right scripts locally. The operational model of "deploy a policy to a server and every Claude Code instance picks it up" is the same model that made centralized linting servers, pre-merge checks, and remote policy engines standard in mature engineering organizations. HTTP hooks are the mechanism that brings Claude Code into that operational model.

One architectural consideration: your HTTP hook endpoint becomes a latency-sensitive dependency. Every synchronous HTTP hook on PreToolUse adds its response time to every tool invocation. If your endpoint takes 200ms to respond, and Claude invokes 50 tools per task, you've added 10 seconds of wall-clock time. Keep your validation logic fast—in-memory rule evaluation, not database queries. Use caching aggressively. Set tight timeouts (the default is 5000ms, but for PreToolUse hooks, 1000ms is a better target). And for any hook where you don't need the response to influence Claude's behavior, use async mode to eliminate the latency entirely.

If your team has more than five developers on Claude Code, start with a single HTTP hook on PreToolUse that logs all Bash commands to a centralized endpoint. Don't try to block anything initially—just collect data on what commands Claude runs across your organization. After a week of data, you'll know exactly which policies to enforce. The audit-first approach builds the case for policy enforcement with evidence, not speculation.
