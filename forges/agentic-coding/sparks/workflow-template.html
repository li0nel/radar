CONTEXT MANAGEMENT

runesleo/claude-code-workflow (539 stars) codifies 3 months of daily session management patterns

The most-starred Claude Code workflow template on GitHub did not emerge from theoretical design. It is the distillation of three months of daily usage, where every pattern was tested against real projects and every convention was refined by actual failure modes. At 539 stars and 66 forks, it represents community consensus on session management more than any other single repository. The template covers three domains: memory management, context engineering, and task routing. Each domain addresses a specific class of failure that daily Claude Code users encounter repeatedly.

The central design philosophy is that manual /compact is categorically superior to auto-compact. This is the most opinionated and most important decision in the template. Auto-compact triggers when the context window fills, which is an arbitrary point determined by token arithmetic, not by the logical structure of the work. You might be mid-analysis when compaction fires, destroying the detailed understanding the agent just built. You might be between two related changes when compaction resets the agent's awareness of the connection between them. Manual compact lets the developer choose the natural break point: after a feature is complete, after a decision is documented, after tests pass. The developer says "this is a good checkpoint" and triggers compact, knowing that everything before this point can be safely summarized because the work is at a logical boundary.

The practical difference is enormous. With auto-compact, developers report sessions where the agent "forgets" mid-task and starts contradicting itself. The compaction summary preserves high-level intent but loses implementation details, intermediate reasoning, and discovered constraints. With manual compact, the developer compacts when the context contains a clean state: completed work summarized, next steps identified, no mid-thought reasoning to lose. The template recommends compacting after every completed task or every 30-40 minutes of active work, whichever comes first.

The workflow template's approach to compact is more nuanced than "always manual." It layers three strategies:

## Compact Strategy (from workflow template)

1. After completing a task: /compact immediately
   - Context is clean: task done, tests pass, changes committed
   - Nothing valuable to lose

2. Before starting a complex task: /compact first
   - Clear out stale context from previous work
   - Start with maximum available context for the new task

3. When context feels heavy but task is incomplete:
   - Write a handoff document FIRST
   - Then /compact
   - Then load the handoff document
   - Agent resumes with clean context + documented state

Strategy 3 is the critical one. It handles the case where you cannot wait for task completion to compact because the context is already degraded. The handoff document captures current state so that compaction does not destroy it. This is manual backup followed by manual compact—the developer controls both the backup content and the compact timing.

Task routing by complexity is the second major contribution. Not every task requires the same model, the same context budget, or the same level of agent autonomy. The template codifies a three-tier routing system:

Simple tasks (typo fixes, small config changes, straightforward implementations with clear specifications): route to Haiku or the fastest available model. These tasks do not benefit from deep reasoning. They need speed and accuracy on well-defined problems. Using Opus for a typo fix is like using a crane to hang a picture frame—it works, but the setup cost exceeds the task cost.

Implementation tasks (new features, refactors, multi-file changes): route to Sonnet. These tasks need good code generation, reasonable context management, and the ability to hold a multi-file mental model. Sonnet handles this well at significantly lower cost and latency than Opus. The template notes that 70-80% of daily coding tasks fall into this category, which means the default model for most work should be Sonnet, not Opus.

Architecture tasks (system design, complex debugging, cross-cutting concerns, decisions with long-term consequences): route to Opus. These tasks benefit from deeper reasoning, better ability to hold large context, and more nuanced judgment about tradeoffs. Architecture decisions made by a less capable model are cheap in the moment and expensive over the project's lifetime. The template recommends Opus specifically for sessions where you are making decisions that will persist across many future sessions.

## Task Routing Heuristic

Ask: "If this task is done wrong, what's the blast radius?"

Low blast radius (< 1 file, easily reversed):
  → Haiku. Speed matters, depth doesn't.

Medium blast radius (multiple files, team-visible):
  → Sonnet. Good balance of capability and cost.

High blast radius (architecture, API contracts, data models):
  → Opus. The cost of a wrong decision exceeds the cost
     of a more capable model by orders of magnitude.

The blast-radius heuristic is the most practical routing decision framework available. It does not require estimating task complexity (which is hard) or classifying task types (which is ambiguous). It requires estimating the cost of failure, which developers are generally good at assessing. A wrong typo fix costs 30 seconds. A wrong database schema change costs days.

Session handoff documents are the third pillar and the template's most directly actionable contribution. The problem: you end a session (by choice, by running out of context, by closing the terminal) and start a new one. The new session knows nothing. Even with CLAUDE.md and memory files, the new session lacks the specifics of where you left off, what you were in the middle of, what you tried that did not work, and what the immediate next step is.

The handoff document is written at the end of every session. It is not optional. It is not "when you remember." It is a mandatory session-closing ritual, like a pilot's shutdown checklist. The template provides the format:

# Session Handoff — [date] [time]

## Completed This Session
- Implemented refresh token rotation in /src/auth/refresh.ts
- Added Redis-backed revocation tracking
- Updated auth middleware to check revocation set
- Tests passing for rotation and revocation flows

## In Progress
- Integration tests for token refresh endpoint (3 of 7 written)
- OpenAPI spec not yet updated for new refresh response shape

## Decisions Made
- Refresh tokens stored in HttpOnly cookies (not localStorage)
  Rationale: XSS protection outweighs slight complexity increase
- 7-day absolute expiry with rotation on every use
  Rationale: balance between security and user experience
- Revocation tracked in Redis SET with TTL matching token expiry
  Rationale: automatic cleanup, no cron job needed

## Tried and Rejected
- Attempted to use JWT blacklist in PostgreSQL — too slow for
  per-request checks at expected scale. Redis SET with O(1)
  lookup is correct for this access pattern.

## Next Session Should
1. Complete remaining 4 integration tests
2. Update OpenAPI spec (see /docs/api/auth.yaml)
3. Add rate limiting to POST /api/v2/auth/refresh (max 10/min)
4. Update MEMORY spoke: memory/auth-strategy.md with new decisions

Every field in this template serves a specific purpose. "Completed" prevents the next session from re-doing work. "In Progress" tells the agent exactly where to resume. "Decisions Made" with rationale prevents the next session from re-opening settled debates. "Tried and Rejected" prevents the next session from going down dead ends that this session already explored—this field alone saves more time than the entire handoff document takes to write. "Next Session Should" is a prioritized task list that eliminates the cold-start problem of "what should I work on?"

The template recommends that handoff documents live in a .claude/handoffs/ directory, named by date and time. The most recent handoff is also symlinked as .claude/handoffs/latest.md. The session-start instruction in CLAUDE.md reads: "At the start of every session, read .claude/handoffs/latest.md and resume from where the previous session left off."

Why this works better than relying on memory spokes alone: memory spokes capture decisions (persistent, topic-organized, long-lived). Handoff documents capture state (ephemeral, session-specific, consumed once). The auth-strategy memory spoke records "we use JWT with refresh tokens in HttpOnly cookies." The handoff document records "integration tests 3/7 done, need to finish the remaining 4 before updating OpenAPI spec." These are fundamentally different types of information with different lifecycles.

The community adoption numbers (539 stars, 66 forks) indicate that the template resonates with daily users, not just people who star repos they will never use. The fork count (66) is the more meaningful metric—forks indicate people who are adapting the template to their own workflows. The ratio of forks to stars (12%) is high for a template repo, suggesting active adoption rather than passive bookmarking.

The template is actively maintained, which matters for a fast-moving ecosystem. Claude Code's capabilities and conventions evolve rapidly—a workflow template from three months ago that has not been updated may reference deprecated features or miss new capabilities. Active maintenance means the template's conventions stay current with the tool.

Limitations worth noting: the template assumes a single-developer workflow. It does not address multi-developer coordination (multiple people running Claude Code sessions on the same codebase), multi-agent orchestration (parallel agents within a single session), or CI/CD integration (triggering agent workflows from pipelines). For these use cases, the template provides a foundation but requires extension. The task routing heuristic also assumes access to multiple models, which depends on your Anthropic plan and configuration. Teams locked to a single model tier get the memory and handoff patterns but not the routing benefit.

The strongest adoption path: take the handoff document template and the manual compact strategy. These two practices, implemented together, eliminate the two most common daily-use frustrations (lost state across sessions, degraded context from auto-compact) with minimal setup cost. Task routing and full memory management can be added incrementally as you build confidence with the workflow.

Relevant if you use Claude Code daily and spend non-trivial time re-establishing context at the start of sessions or recovering from auto-compaction mid-task.
