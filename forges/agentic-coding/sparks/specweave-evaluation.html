WORKFLOW DESIGN

SpecWeave adds persistent quality gates across sessions and tools

SpecWeave (npm install -g specweave) is a spec-first behavior enforcement layer that sits between your intent and your AI coding tools. It works with Claude Code, Cursor, Copilot, and Codex. The core thesis is simple and correct: if you cannot express a quality standard as a machine-readable specification, that standard will not survive contact with agentic workflows at scale. SpecWeave makes specifications the durable artifact rather than the conversation history, the CLAUDE.md file, or the developer's memory.

The problems it solves are real and measurable. Session amnesia: session 47 does not know what session 46 decided about error handling patterns, API response shapes, or test coverage thresholds. You can write it in CLAUDE.md, but CLAUDE.md is instructions, not enforcement. Nothing stops an agent from ignoring instructions when context gets tight or when the model decides the instruction is less relevant than the immediate task. SpecWeave turns decisions into gates. The agent cannot merge an increment that violates a gate. This is categorically different from "please follow these guidelines."

Standards lottery: without enforcement, the quality of any given session depends on which model you hit, how much context remains, how well-written the prompt was, and whether Mercury is in retrograde. Run the same task ten times across ten sessions and you get ten different quality levels. Some sessions produce pristine code with comprehensive tests. Others produce code that technically works but violates six team conventions. SpecWeave normalizes this. The gate is the gate. It does not care whether the session was lucky or unlucky.

Coordination failure: when parallel agents work on related features, they make conflicting decisions. Agent A adds a new error type; Agent B adds a different error type for the same condition. Agent A picks camelCase for the new API field; Agent B picks snake_case. Without a shared enforcement layer, you discover these conflicts at code review time, which means a human is doing the coordination work that the system should handle.

SpecWeave's agent swarm capability addresses coordination directly. Each agent owns an isolated increment. The increment is a bounded unit of work with explicit inputs, outputs, and quality gates. No two agents modify the same files simultaneously. When an agent completes its increment, the result passes through the spec gates before being integrated. Conflicts are caught at the gate, not at review. The project has validated this approach across 635+ increments of self-building—SpecWeave is built using SpecWeave, which is the strongest possible dogfooding signal.

The token reduction claim deserves scrutiny. SpecWeave reports 70%+ token reduction via its plugin architecture. The mechanism is straightforward: instead of stuffing context into the conversation window (which the agent then ignores once it scrolls past the attention horizon), SpecWeave externalizes specifications into files that are loaded on demand. The plugin architecture means the agent only loads the specs relevant to the current increment, not the entire specification corpus. This is not magic; it is the same principle behind hub-and-spoke memory patterns, applied to specification enforcement. The 70% figure likely measures total tokens across a multi-session feature build, comparing "everything in CLAUDE.md" against "specs loaded per increment." Credible, but environment-dependent.

The LSP code intelligence integration is genuinely impressive. SpecWeave claims 198x faster than grep for code navigation. This is plausible because LSP provides semantic understanding—it knows that getUserById in file A is the same function referenced in file B, while grep just matches strings. For an agent deciding where to make changes, semantic navigation prevents entire classes of errors: modifying the wrong function with the same name, missing call sites, breaking interfaces without updating implementations. The LSP integration turns SpecWeave from "spec enforcement" into "spec enforcement with code understanding," which is a meaningful architectural advantage over homegrown solutions.

Living documentation is the other differentiator. Specifications auto-update via hooks when code changes. This inverts the usual relationship where docs lag code. In a SpecWeave project, the specification is the source of truth, and the code must conform to it. When the specification changes (through a deliberate process, not an agent whim), the documentation updates automatically. When the code changes, the hooks verify spec compliance and update any derived documentation. This is how specifications should work but almost never do in practice.

The configuration system makes enforcement concrete:

testing:
  defaultTestMode: "TDD"
  coverageThreshold: 80
quality:
  grillRequired: true
  maxComplexity: 15
  requiredPatterns:
    - error-handling
    - input-validation

grillRequired: true is the key line. "Grill" in SpecWeave terminology is the verification phase where the completed increment is tested against its specification. When grill is required, the agent cannot declare an increment complete without passing verification. This is the difference between "we encourage testing" and "you cannot ship without tests." Every team says they value quality. SpecWeave teams enforce it mechanically.

The practical decision framework for when to use SpecWeave versus lighter approaches follows a clear complexity gradient. Simple bug fixes (typos, off-by-one errors, straightforward patches): just talk to the AI directly. SpecWeave overhead exceeds the benefit. Small features (single-file additions, minor API endpoints): write a brief plan, then implement. A CLAUDE.md instruction is sufficient. Medium features (multi-file changes, new subsystems): brainstorm the approach, create a SpecWeave plan, then run auto-implementation against the spec. This is where SpecWeave starts earning its keep because the spec prevents drift across the multiple files. Large refactors (architectural changes, cross-cutting concerns): use agent teams with isolated increments. This is SpecWeave's core value proposition—without spec-enforced coordination, multi-agent refactors produce chaos.

The tradeoffs are real and should not be minimized. SpecWeave is heavyweight for simple tasks. The ceremony of creating specifications, defining increments, and configuring quality gates takes time. If your project is small or your changes are simple, this ceremony costs more than it saves. It is another tool in the chain, which means another point of failure, another thing to update, another learning curve for new team members. The specification language itself requires investment—writing good specs is a skill, and writing specs that are precise enough for machine enforcement but flexible enough for real-world code is harder than it looks.

Compared to rolling your own enforcement with Claude Code skills and hooks: homegrown solutions are cheaper to start, perfectly tailored to your workflow, and require no external dependency. But they lack the LSP integration, the increment isolation for agent swarms, the living documentation, and the 635+ increments of battle-testing. If you need multi-agent coordination on a large codebase, SpecWeave solves problems that are genuinely hard to solve with hooks alone. If you need simple quality gates on a small project, a well-written PreToolUse hook accomplishes 80% of the value at 10% of the complexity.

The self-building track record (635+ increments) is the strongest argument for SpecWeave's viability. It is also the strongest argument for its complexity ceiling—a tool that requires 635 increments to build itself is not a tool optimized for simplicity. Evaluate accordingly based on your project's actual coordination needs, not aspirational ones.

Relevant if you are running multi-session or multi-agent workflows where quality drift between sessions is a measurable problem, or if you need agent swarm coordination beyond what hooks and CLAUDE.md can enforce.
