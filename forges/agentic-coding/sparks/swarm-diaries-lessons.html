Orchestration
Microsoft's Swarm Diaries document the failure modes of uncoordinated multi-agent development

The Azure Apps team at Microsoft ran a controlled experiment in Q4 2025: five AI coding agents working simultaneously on a single codebase for a two-week sprint. The team published their findings as "Swarm Diaries" on the Microsoft Tech Community blog. The results are the most detailed public documentation of what actually happens when you throw multiple agents at a real production codebase without adequate coordination infrastructure. The short version: it works brilliantly in the narrow case and fails catastrophically in the general case, and the boundary between the two is more precise than you'd expect.

The experiment used five instances of GitHub Copilot Workspace (the agent-mode version, not the autocomplete) on a mid-size Azure service: approximately 180,000 lines of C# across 40 projects in a monorepo. Each agent was assigned a high-level task from the sprint backlog. The tasks were chosen to represent a realistic mix: two feature implementations, one refactoring, one test coverage improvement, and one documentation update. The agents were given access to the full repository and could create branches, run tests, and submit PRs. No coordination protocol was established between agents—this was deliberate, as the team wanted to observe the failure modes of uncoordinated parallel work.

Failure mode one: merge conflicts on shared files. This was the most frequent and most predictable failure. Three of the five agents modified the dependency injection configuration file (Startup.cs in the older codebase, Program.cs in the newer pattern). Each agent added service registrations for its feature. Each agent's change was correct in isolation. When the PRs were merged sequentially, the second and third PRs had textual merge conflicts. The merge conflicts themselves were trivial to resolve (add all three service registrations), but they required human intervention, broke the automated merge pipeline, and invalidated the CI results on the conflicting PRs. The agents had no way to know that another agent was modifying the same file, no way to sequence their changes, and no way to resolve the conflict themselves.

The team's analysis identified the root cause precisely: shared infrastructure files. In any codebase, there are files that are modified by many features: DI configuration, routing tables, database migration runners, shared type definitions, and central configuration files. These files are not "owned" by any feature—they're cross-cutting. A single developer working on multiple features touches these files once and keeps them consistent. Five agents working in parallel each touch these files independently and produce five inconsistent versions.

The proposed mitigation, which the team tested in the second week: designate one agent as the "infrastructure agent" with exclusive write access to shared files. Other agents declare their infrastructure needs (e.g., "I need a service registration for IOAuthService") in a structured format, and the infrastructure agent applies all changes to shared files in a coordinated sequence. This eliminated merge conflicts entirely but introduced a serialization bottleneck—the infrastructure agent became a choke point that other agents waited on. The team estimated this reduced overall throughput by 15-20% compared to the theoretical parallel maximum, but the reduction was worth it because it eliminated the 100% merge failure rate on shared files.

Failure mode two: inconsistent architectural decisions. This was subtler and more damaging. The two feature agents independently chose different patterns for the same architectural concern. One agent implemented error handling with a Result monad pattern (returning Result<T, Error> from service methods). The other agent implemented error handling with exceptions (throwing custom exception types and catching them in middleware). Both patterns are valid. Both agents had reasonable justifications based on the patterns they observed in the existing codebase (which, like most real codebases, had a mix of both patterns). The result was a codebase where the new OAuth feature used Result types and the new notification feature used exceptions. Not a merge conflict. Not a compilation error. Not a test failure. A silent architectural inconsistency that would compound over time.

This failure mode is particularly insidious because it's invisible to automated quality gates. Tests pass. The build succeeds. Code review might catch it, but only if the reviewer examines both PRs in the context of each other—which is unlikely when they arrive as separate, independently-authored PRs. The inconsistency only becomes apparent later when a developer works on a feature that touches both modules and has to understand two different error handling conventions in the same codebase.

The mitigation the team proposed: architectural decision records (ADRs) that agents must read before starting work. The ADRs declare the canonical patterns for common concerns (error handling, logging, data access, authentication). Each agent's system prompt includes a directive to check ADRs before making architectural choices. In the second-week test, this reduced architectural inconsistencies significantly but didn't eliminate them—agents sometimes made decisions that fell outside the scope of existing ADRs, and there was no mechanism for an agent to propose a new ADR when it encountered a novel concern.

Failure mode three: cascading test failures. The refactoring agent modified an internal utility class that was used by 23 test files. The modification was correct—the utility's public API was preserved—but the implementation change altered a timing-dependent behavior that 6 integration tests relied on (tests that were, in hindsight, testing an implementation detail rather than a contract). The refactoring agent's own tests passed because it ran unit tests for the refactored module. The 6 failing integration tests belonged to other modules and were not in the refactoring agent's test scope.

When the feature agents ran their tests later, they encountered failures in tests they hadn't modified, couldn't understand why, and attempted to "fix" the tests by adjusting expected values or adding waits—masking the real issue rather than identifying that the refactoring agent had changed upstream behavior. The human team spent four hours untangling this: reverting the test "fixes," understanding the refactoring's actual behavior change, deciding whether the behavior change was intentional, and updating the integration tests to test the contract rather than the implementation.

The cascading test failure mode is the most expensive because it compounds. Agent A changes behavior. Agent B sees test failures and "fixes" them by adjusting tests to match the new behavior, even if the new behavior is a bug. Agent C depends on the test suite for confidence and merges changes that are correct against the (now-wrong) tests. By the time a human investigates, three agents' work products are entangled with the original behavior change, and the root cause is buried under layers of compensating changes.

The mitigation: no agent should modify tests that it didn't create or that aren't in its domain. If an agent encounters a failing test outside its domain, it should stop and report the failure rather than attempting to fix it. This is easy to enforce with file ownership rules (the same domain glob mechanism that Agent Teams uses) but requires the coordination infrastructure to support it. The uncoordinated experiment had no such enforcement, which is why the failure cascaded.

Now, the wins. On the three workstreams that were genuinely independent—the test coverage improvement (touching only test files, no source changes), the documentation update (touching only docs/), and one of the feature implementations (a new, isolated microservice with no shared files)—the agents delivered 3-5x throughput compared to the estimated single-developer timeline. The documentation agent completed in 3 hours what would have taken a developer a full day. The test coverage agent wrote 147 new tests in 6 hours. The isolated feature agent delivered a complete microservice with tests in 8 hours.

The 3-5x multiplier is real but conditional. It requires that the workstreams share no files, make no architectural decisions that affect each other, and have no test dependencies. In the Azure Apps codebase, approximately 30% of sprint tasks met these criteria. The other 70% had at least one dependency that required coordination. This ratio is consistent with other reports: multi-agent parallelism delivers its theoretical throughput on embarrassingly parallel workstreams and delivers negative throughput (slower than single-agent due to conflict resolution overhead) on entangled workstreams.

The key findings, distilled to actionable rules:

Rule 1: Single owner per shared file. Any file that appears in more than one agent's potential edit set must be assigned to exactly one agent. Other agents request changes through the owning agent. This eliminates merge conflicts at the cost of serialization on shared files.

Rule 2: Explicit interface contracts before parallel work begins. Agents must agree on shared interfaces (API types, function signatures, database schemas) before any implementation starts. The contracts are written to files that all agents can read but only the owning agent can modify. Changes to contracts require a coordination step, not unilateral action.

Rule 3: Genuinely independent units. If you cannot define the unit of work such that it touches no files outside its domain and has no behavioral dependencies on other units, it is not ready for parallel execution. Decompose further or execute sequentially.

Rule 4: Never fix tests you don't own. When an agent encounters a failing test outside its domain, it stops and reports. It does not attempt to fix, adjust, or work around the failure. This prevents the cascading fix problem that was the most expensive failure mode in the experiment.

Rule 5: Checkpoint-merge frequently. Don't let agents work for 8 hours and then merge. Merge every 1-2 hours into an integration branch and run the full test suite. Catch conflicts and inconsistencies early when the blast radius is small.

For smaller teams and solo developers, the implications are clear. Multi-agent development is not a general-purpose speedup. It's an optimization for a specific workload pattern: genuinely independent tasks with clear boundaries. If you're a solo developer, the tasks most likely to benefit are: test writing (independent of source changes), documentation (independent of implementation), dependency updates (mechanical, file-scoped), and new modules that don't touch existing code. Tasks that don't benefit: refactoring existing code (touches shared state), feature work that spans multiple modules (requires coordination), and any work that modifies shared infrastructure files.

The Swarm Diaries' most valuable contribution is the empirical evidence that coordination overhead is not zero. The multi-agent narrative in the industry tends to focus on the theoretical throughput multiplier (N agents = Nx throughput) without accounting for the coordination cost. Microsoft's data shows that coordination cost scales superlinearly with the number of agents and with the degree of coupling between their workstreams. At five agents with 70% coupled workstreams, the coordination cost consumed most of the theoretical throughput gain. The net improvement was approximately 1.8x over a single agent—real, but far from the 5x the setup implied.

The honest assessment: multi-agent development is not a silver bullet. It's a tool with a narrow applicability window that widens as coordination infrastructure improves. The tools (ACPX, Agent Teams, cmux) are improving rapidly. The decomposition practices (domain splits, interface contracts, file ownership) are being documented and formalized. The Swarm Diaries are valuable because they tell you exactly what fails and why, so you can invest in the right mitigations instead of discovering the failure modes in production.

Before attempting multi-agent work on any feature, run the Swarm Diaries checklist: Can you define genuinely independent units? Can you assign single owners to every shared file? Can you write interface contracts before implementation starts? If any answer is no, stay single-agent for that feature. The 3-5x gains are only available on the 30% of tasks that pass this checklist.
